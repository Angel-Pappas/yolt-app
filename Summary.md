# Summary

This file exists so a fresh session can get up to speed on this app quickly. Keep it current: update the relevant section whenever a major feature is added, removed, or changed.

## General description

Yolt is a personal financial transactions tracker, built for a single user (the app owner only — no multi-tenant/sharing features are planned). It supports logging typed transactions (Income, Expense, or Transfer between wallets — date, entity, wallet(s), description, net amount, VAT) behind email/password authentication, plus supporting lists of Entities, Wallets, and VAT rates, and a Taxes overview. Planned future features include financial projections, automatic generation of recurring transactions, and richer tax handling (multiple tax types, input/output VAT netting), plus whatever else comes up as the app grows.

## Main sections of the app

- **Authentication** — Signup, login, and logout. Email/password via Supabase Auth, including the email-confirmation flow. Pages: `/login`, `/signup`, `/auth/confirm` (confirmation link handler), `/auth/auth-code-error` (expired/invalid link fallback).
- **Home** (`/`) — Minimal landing page after login. Placeholder content for now.
- **Transactions** (`/transactions`) — Full CRUD on transactions (date, type, entity, wallet(s), description, net, VAT rate), all backed by RLS-scoped Server Actions (`actions.ts`: `addTransaction`, `updateTransaction`, `deleteTransaction`).
  - **Add**: a text "Add" button opens a modal (`transaction-modal.tsx`).
  - **Edit**: no button — clicking anywhere on a transaction's row (`transaction-row.tsx`, a client component) opens the same edit modal, pre-filled.
  - **Delete**: a trash-icon button per row (`delete-transaction-button.tsx`, `src/components/icons.tsx` for the icon — inline SVG, no icon library dependency); its click handler (and the dialog's) calls `stopPropagation()` so it doesn't also trigger the row's edit-open behavior. Delete is a **soft delete** — see the Database section below. Nothing is ever hard-deleted through the app.
  - Both modal contexts share one dialog implementation (`transaction-form-dialog.tsx`, native `<dialog>`, unique field ids via `useId`).
  - **Type** (`income` / `expense` / `transfer`) is picked first in the form and drives which other fields render — see the "Transaction types" subsection below.
  - **Entity** field (Income/Expense only) is a client-side-filtered search-and-select (`entity-combobox.tsx`) over the already-loaded entity list, with a hidden `entity_id` input and a "+ Add" button that opens the entity-add dialog (`entities/entity-form-dialog.tsx`) as a **sibling** of the transaction `<form>`, not nested inside it — nesting a `<dialog><form>` inside another `<form>` is invalid HTML and breaks under SSR. No inline creation from the combobox itself; the user must pick the newly-created entity from the search afterward.
  - **Wallet** and **VAT** (Income/Expense) are plain `<select>` dropdowns (populated from active wallets / active VAT rates). Transfer shows **From wallet** / **To wallet** selects instead (both required, and must differ — checked both client-side, for a quick inline error, and server-side/DB-side, since that's the actual source of truth).
  - **VAT Amount** and **Total** are calculated, read-only (live-updated client-side as Net/VAT change for display, but never trusted from the client): `vat_amount` is computed and stored server-side in the Server Action from `net × the selected rate's current percentage` at save time (so editing a VAT rate's percentage later doesn't retroactively change past transactions' stored VAT amount). `Total` (`net + vat_amount`) is never stored at all — always computed on read via `computeTotal()` in `src/lib/format.ts`. Transfers have no VAT, so `vat_amount = 0` and `Total = net` for them.
  - **Modals close on Esc (native, free) and on clicking the backdrop.** The backdrop-click handler checks that the click landed exactly on the `<dialog>` element itself (`e.target === e.currentTarget`) — which only happens for the backdrop, *because* each dialog's padding/border/background live on the inner `<form>`, not the `<dialog>`. If a future dialog puts padding directly on the `<dialog>`, clicking that padding will wrongly count as "outside" and close it — keep visual chrome on an inner wrapper.
  - **Transaction types**: `Income` and `Expense` each affect exactly one wallet (via `wallet_id`) — income adds `net + vat_amount` to its balance, expense subtracts it. `Transfer` moves money between two of the user's own wallets with no entity/VAT: `wallet_id` is the "from" side (subtracts `net`), `to_wallet_id` is the "to" side (adds `net`) — see Database section for the DB-level constraint enforcing this split, and the Wallets section for how a transfer shows up in both wallets' ledgers from one single row (not two linked rows — much simpler for edit/delete, since there's nothing to keep in sync).
  - **Filtering, sorting, pagination**: all state lives in the URL's search params (`q`, `type`, `entity`, `wallet`, `vat`, `from`, `to`, `sort`, `dir`, `page`) via a shared `useTransactionParams()` hook (`use-transaction-params.ts`) — the idiomatic App Router pattern: bookmarkable/shareable, and the Server Component (`page.tsx`) reads them directly and re-queries, no client-side data duplication. `parseFilters()`/`parseSort()` in `page.tsx` validate every param (enum/UUID/date-format) before anything reaches `getActiveTransactions()` — an invalid value passed straight to `.eq`/`.or` on a uuid column would error the whole query, not just match nothing, so invalid values are silently dropped instead (treated as "no filter").
    - **`transaction-filters-bar.tsx`**: search box (description only — Entity/Wallet/VAT have their own per-column filter instead, see below) with local state + a 300ms debounce so typing doesn't trigger a server round-trip per keystroke (re-synced from the URL on external changes like Clear filters by adjusting state during render rather than in a `useEffect` — see Notes to myself), plus From/To date range with quick presets (this month/last month/this year/all time) and a Clear filters button.
    - **`transaction-table-header.tsx`**: every column header is a sort button (▲/▼ indicator on the active one; click again to flip direction) — Type/Entity/Wallet/VAT headers additionally carry their own filter `<select>` right in the header cell. Wallet filtering matches either side of a transfer (`wallet_id` or `to_wallet_id`), same as the wallet ledger.
    - **Sorting** (`sortTransactions()` in `queries.ts`) and **pagination** (25/page) both happen in JS, against the *entire* filtered result fetched from Supabase — not via PostgREST `.order()`/`.range()`. Deliberate: sorting by Entity/Wallet means sorting by a joined table's `name`, which is real friction to get right via PostgREST's embedded-resource ordering; sorting in JS after a plain unordered fetch sidesteps that entirely and is trivial to get correct. Fine at this app's single-user scale — revisit (server-side sort + range-based pagination) if the transaction count ever gets large enough that fetching the whole filtered set every page load starts to matter.
    - Changing any filter or sort resets pagination back to page 1 (`setFilterParams` in the shared hook always clears `page`); only `setPage` (Previous/Next) touches the page param alone.
    - Deliberately **not built**: a totals/summary bar (tried, then explicitly removed — not useful for this app) and CSV export (out of scope for now).
- **Entities** (`/entities`) — Full CRUD list of the counterparties transactions are made with (suppliers, customers, the state, etc.). Fields: name (required), VAT number (optional). Same soft-delete CRUD pattern as Transactions (row click opens edit modal, trash icon soft-deletes).
- **Wallets** (`/wallets`) — Full CRUD list of where money is held (bank accounts, cash, etc.). Fields: name only. No stored balance column — the list page shows each wallet's **current balance computed live** (`getWalletBalances()` in `wallets/queries.ts`), so it can never drift out of sync as transactions are added/edited/deleted/restored. Since the wallet name is a link (to the ledger view below) rather than the whole row, edit uses a pencil-icon button (`wallet-row.tsx`) instead of the row-click pattern used elsewhere.
  - **Ledger view** (`/wallets/[id]`) — a per-wallet transaction timeline with a running balance column, computed by walking that wallet's transactions in chronological (oldest-first) order (`getWalletLedger()`), matching the rest of the app. A transfer shows up in **both** wallets' ledgers (it's one row, matched by `wallet_id = this wallet OR to_wallet_id = this wallet`), signed from each wallet's own point of view and labeled the same "X → Y" way as the Transactions page (the actual from/to wallet names, not "Transfer" + an arrow) — so the current balance is the **last** entry's running balance, not the first.
- **Taxes** (`/taxes`) — Read-only. Currently shows a single live-computed total (`SUM(vat_amount)` across active transactions, via `getTotalVat()`). Not a CRUD list — there's nothing to store, it's purely an aggregate over transactions. More tax types (and more intricacies — input/output VAT netting, filing periods, etc.) are expected here later.
- **Options** (`/options`) — Account settings (email/password, unchanged) plus a **VAT rates** section: full CRUD list (name, rate %) that populates the VAT dropdown on the transaction form. Same soft-delete pattern; files `vat-rate-*.ts(x)` alongside the existing `actions.ts`.
- All pages above share a common app shell (`src/app/(app)/layout.tsx`, a route group) with a nav bar (Home / Transactions / Entities / Wallets / Taxes / Options / Log out).

## Technical sections

- **Framework**: Next.js 16 (App Router, Turbopack), TypeScript, Tailwind CSS v4, ESLint. Scaffolded with `create-next-app`.
- **Data/validation libs installed for future use**: `@tanstack/react-query`, `zod` (not yet wired into any feature).
- **Auth/session plumbing**: `@supabase/ssr` with three client setups:
  - `src/lib/supabase/client.ts` — browser client
  - `src/lib/supabase/server.ts` — server client (Server Components/Actions)
  - `src/lib/supabase/proxy.ts` + `src/proxy.ts` — session-refresh and route protection. Next.js 16 renamed "middleware" to "proxy" (file `proxy.ts`, function `proxy`) — this app follows that convention. Auth check uses `supabase.auth.getClaims()` (not `getSession()`/`getUser()`), per current Supabase guidance.
  - Unauthenticated visitors are redirected to `/login` for any route except `/login`, `/signup`, and `/auth/*`.
- **Page structure**: `src/app/(app)/` is a route group holding the authenticated app shell — `layout.tsx` (nav bar), `page.tsx` (Home, `/`), `transactions/page.tsx` + `transactions/actions.ts` (`/transactions`), `options/page.tsx` (`/options`). Route groups (parens in the folder name) don't add a URL segment. `/login`, `/signup`, and `/auth/*` live outside this group (no nav bar).
- **Database**: Supabase Postgres. Project `yolt-app`, ref `mzfxfweljbfvyqlhvmzr`, region `eu-west-1`, free tier, org "Angel-Pappas's Org" (`yowsydnsxjwcujsjwtji` — same org as the Exalted-Character-App project).
  - Table `public.transactions`: `id` (uuid pk), `user_id` (uuid, defaults to `auth.uid()`), `date` (date), `description` (text), `type` (text, `check (type in ('income','expense','transfer'))`), `net` (`numeric(12,2)`, `check (net >= 0)` — always a non-negative magnitude; `type` determines direction, the number itself never carries a sign), `entity_id` (uuid, nullable FK → `entities.id`, `on delete set null`), `wallet_id` (uuid, **not null**, FK → `wallets.id` — the affected wallet for income/expense, the "from" wallet for a transfer), `to_wallet_id` (uuid, nullable, FK → `wallets.id` — only set for transfers, the "to" wallet), `vat_rate_id` (uuid, nullable FK → `vat_rates.id`, `on delete set null`), `vat_amount` (`numeric(12,2)`, not null default 0, `check (vat_amount >= 0)` — computed and stored server-side at save time, see Transactions section above for why), `created_at`, `is_deleted` (default false), `deleted_at` (nullable). There is **no `amount` column anymore** — it was replaced by `net` + `vat_amount` (with `Total` derived, never stored). RLS enabled, all policies scoped to `auth.uid() = user_id`.
    - `wallet_id`'s and `to_wallet_id`'s FKs deliberately do **not** use `on delete set null` (unlike `entity_id`/`vat_rate_id`) — a hard-delete of a wallet that still has transaction history should fail loudly (Postgres default `NO ACTION`/restrict), not silently null out `wallet_id` (which would violate its `NOT NULL`) or silently turn a transfer into a broken half-transfer.
    - `transactions_type_fields_check` CHECK constraint ties `type` to which other fields are legal: `transfer` requires `to_wallet_id` set and different from `wallet_id`, plus `entity_id`/`vat_rate_id` both null and `vat_amount = 0`; `income`/`expense` require `to_wallet_id` null and `vat_rate_id` set. This is enforced at the DB level, not just in the Server Action, so it can't be bypassed by a bug in the app layer.
  - Tables `public.entities`, `public.wallets`, `public.vat_rates`: same shape/conventions — `id`, `user_id` (default `auth.uid()`), `name`, plus one extra field each (`vat_number` for entities, `rate numeric(5,2)` for vat_rates; wallets has no extra field), `created_at`, `is_deleted`, `deleted_at`. Same RLS pattern as transactions.
  - **Soft delete**: the app's "Delete" action does `update is_deleted = true, deleted_at = now()`, never a real `DELETE` — on **every** table (transactions, entities, wallets, vat_rates). Filtering out `is_deleted = true` rows is done **only at the app-query level**, via a single shared helper per table (`getActiveTransactions()`, `getActiveEntities()`, `getActiveWallets()`, `getActiveVatRates()`) — see the important RLS gotcha in Notes to myself: each table's SELECT RLS policy deliberately does *not* filter on `is_deleted`, because it can't without breaking the soft-delete UPDATE itself. There is **no DELETE policy** on any of these tables (dropped/never added deliberately) — the app has no way to hard-delete a row, by design. Only the user (owner) can ask for a permanent delete or a restore, done by directly running SQL via the Supabase MCP (`execute_sql`/`apply_migration`), not through any in-app feature.
  - Editing a `vat_rates` row's `name`/`rate` only affects the dropdown label and future transactions — it never retroactively changes `vat_amount` on transactions that already used that rate, since that value was snapshotted at save time.
- **Hosting/deploy**: Vercel, connected to GitHub repo [Angel-Pappas/yolt-app](https://github.com/Angel-Pappas/yolt-app), auto-deploys on push to `main`. Env vars (`NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY`) are set directly in the Vercel dashboard (mirrors `.env.local`, which is gitignored; `.env.example` is committed as a template).

## Formatting conventions

Greek-style formatting is used everywhere the app displays dates/numbers: dates as `dd/mm/yyyy`, numbers with `.` as the thousands separator and `,` as the decimal separator, always rounded to exactly 2 decimals. Shared helpers live in `src/lib/format.ts` (`formatDate`, `formatAmount`, `computeTotal`) — use these instead of ad-hoc formatting/arithmetic whenever displaying a date or amount anywhere in the app. Native HTML form inputs (`<input type="date">`, `<input type="number">` in the add-transaction modal) are exempt — their `value` must stay ISO/period-decimal per the HTML spec, only their on-screen picker rendering is browser/OS-locale-dependent, and that isn't something the app controls.

## Notes to myself

- This is a single-user app. Don't add roles/teams/sharing/multi-tenant complexity unless explicitly asked — but keep RLS correct regardless, it's good practice regardless of user count.
- **Build this properly, not as a scrappy personal script** (user's explicit direction, ~2026-07): the user is about to import ~750 historical transactions and expects the app to keep growing well beyond that. Default to the production-grade version of a decision over the quick/simple one, even when the simple version would technically work at today's data volume — don't wait to be asked twice to revisit a "fine for now" shortcut once real scale shows up. This doesn't license unrequested features/abstractions (still single-user, still no premature complexity) — it's about engineering robustness (correctness, performance, maintainability), not scope creep.
- `sortTransactions()` in `transactions/queries.ts` fetches the *entire* filtered result and sorts/paginates in JS rather than via PostgREST `.order()`/`.range()` (see that file's docstring for why — Entity/Wallet sorting means sorting by a joined table's name). This was explicitly scoped as "fine at this app's scale, revisit once data volume grows" — the ~750-transaction import is that trigger. Move to server-side sorting/pagination (and add DB indexes on the filtered/sorted columns: `date`, `type`, `wallet_id`, `entity_id`, `vat_rate_id`) before or shortly after that import lands, not after performance actually becomes a visible problem.
- Money is stored as Postgres `numeric(12,2)`, which is exact fixed-point decimal — no floating-point rounding risk. Don't "fix" this into cents/bigint, it's already correct as-is.
- Derived/aggregate values (wallet balances, the VAT total on the Taxes page, `Total` on transactions) are **never stored** — always computed live from their inputs at read time. This was a deliberate choice over a maintained running counter: no increment/decrement logic to get right on every create/edit/delete/restore, and it can't ever drift. Keep following this for any future aggregate (don't add a stored counter without discussing it first).
- A `<dialog>` containing a `<form>` must never be nested *inside* another `<form>` — nested `<form>` elements are invalid HTML and get mangled by the browser's parser during SSR (the inner `<form>` tag is silently dropped since a form element is already open), causing a hydration mismatch. Where a modal-within-a-modal is needed (e.g. the "+ Add" entity button on the transaction form, which opens the entity-add dialog), render the inner `<dialog>` as a **sibling** of the outer `<form>` — a child of the outer `<dialog>`, not of the inner `<form>` — and trigger it via a ref from a plain button inside the outer form. See `transactions/transaction-form-dialog.tsx`.
- Money-affecting `numeric` columns (`net`, `vat_amount`, `rate`) come back from the Supabase JS client as **strings**, not numbers (same as the pre-existing `amount` column did) — always `Number(...)` them before doing arithmetic, same as `formatAmount`/`computeTotal` already do.
- `net` and `vat_amount` are always non-negative magnitudes (DB-enforced via CHECK constraints) — direction/sign is never stored in the number itself, only implied by `type` (and, for transfers, by which wallet role a given wallet plays). Don't reintroduce signed `net` values; compute the signed effect at the point of use (wallet balance/ledger) instead, same as `wallets/queries.ts` already does.
- Native `<dialog>` centering: browsers center a modal `<dialog>` via `margin: auto` on its `position: fixed; inset: 0` box, but Tailwind's Preflight resets all margins to `0`, which strands every dialog top-left unless overridden. Fixed once, globally, in `src/app/globals.css` (`dialog:modal { margin: 1.5rem auto auto; }`) rather than per-component — any new dialog gets this for free.
- All 4 dialog-based CRUD forms (transactions, entities, wallets, VAT rates) share the same "backdrop-click closes the modal" implementation independently (no shared component) — if a 5th one is added later, copy the `handleBackdropClick` pattern from any of them rather than reinventing it, and remember the padding-on-the-form-not-the-dialog requirement noted in the Transactions section above.
- Filter/search/table-view state belongs in URL search params, not client component state — that's the established App Router pattern here (see Transactions' filter bar) and should be followed for any future filterable list, not just transactions.
- Don't call `setState` synchronously inside a `useEffect` just to sync local state from a prop/URL value that changed — ESLint (`react-hooks/set-state-in-effect`) flags this as a cascading-render anti-pattern. Instead adjust state directly during render: compare the incoming value against a "previous value" state var, and if it changed, call both setters right there in the render body (React bails out and re-renders before committing, so there's no extra visible frame). See `transaction-filters-bar.tsx`'s search box for the pattern.

- **Postgres RLS gotcha (learned the hard way, don't repeat it on future tables)**: an `UPDATE`'s resulting row must satisfy the table's `SELECT` policy too, not just the `UPDATE` policy's own `WITH CHECK` — this is enforced unconditionally, regardless of what the `UPDATE` policy's check says. So a SELECT policy that filters out `is_deleted = true` makes it *structurally impossible* for a user to ever set `is_deleted = true` on their own row (the resulting row would be invisible to their own write, which Postgres blocks outright — this produces the exact error `new row violates row-level security policy`). The fix: for any table with a soft-delete column, the SELECT RLS policy must stay unrestricted on that column (just `auth.uid() = user_id`, no `is_deleted` condition) — do the "hide deleted rows" filtering only in a shared app-level query helper. If accounts/categories/etc. get soft-delete later, follow the same pattern: don't add the deleted-flag condition to the table's SELECT policy.
