# Summary

This file exists so a fresh session can get up to speed on this app quickly. Keep it current: update the relevant section whenever a major feature is added, removed, or changed.

## General description

Yolt-App is a personal financial transactions tracker, built for a single user (the app owner only — no multi-tenant/sharing features are planned). It supports logging typed transactions (Income, Expense, or Transfer between wallets — date, invoice date, entity, category, wallet(s), description, net amount, VAT) behind email/password authentication, plus supporting lists of Entities, Wallets, and a growing **Lists** section of reference/lookup data (VAT rates, Categories, more to come), and a Taxes section with an auto-computed monthly VAT breakdown. Planned future features include financial projections, automatic generation of recurring transactions, more tax types beyond VAT, a real notifications system (currently a placeholder — see Options/Account section below), plus whatever else comes up as the app grows.

## Main sections of the app

- **Authentication** — Signup, login, and logout. Email/password via Supabase Auth, including the email-confirmation flow. Pages: `/login`, `/signup`, `/auth/confirm` (confirmation link handler), `/auth/auth-code-error` (expired/invalid link fallback).
- **Home** (`/`) — Minimal landing page after login. Placeholder content for now.
- **Transactions** (`/transactions`) — Full CRUD on transactions (date, type, entity, category, wallet(s), description, net, VAT rate), all backed by RLS-scoped Server Actions (`actions.ts`: `addTransaction`, `updateTransaction`, `deleteTransaction`).
  - **Add**: a text "Add" button opens a modal (`transaction-modal.tsx`).
  - **Edit**: no button — clicking anywhere on a transaction's row (`transaction-row.tsx`, a client component) opens the same edit modal, pre-filled.
  - **Delete**: a trash-icon button per row (`delete-transaction-button.tsx`, `src/components/icons.tsx` for the icon — inline SVG, no icon library dependency); its click handler (and the dialog's) calls `stopPropagation()` so it doesn't also trigger the row's edit-open behavior. Delete is a **soft delete** — see the Database section below. Nothing is ever hard-deleted through the app.
  - **Reconcile** (2026-07, `reconcile-button.tsx`/`reconcile-modal.tsx`, `is_reconciled` column) — a check-circle button that opens a **reduced** edit modal: only Date, Amount, and Wallet(s) are editable (no description/entity/VAT rate). Built ahead of automatic recurring-transaction generation (planned, not built yet — see Notes to myself): a future transaction gets auto-created with a projected date/amount/wallet, and once it actually happens the user opens Reconcile to correct whichever of those three drifted from the projection, then the row is marked reconciled regardless of whether anything actually changed — reconciling is itself the record that the user checked it, not just a change-tracking flag. The button lights up (`text-success`, a dedicated bright green distinct from Income's semantic green — see Design system's Tokens bullet for why) once `is_reconciled` is true. `reconcileTransaction()` (`actions.ts`) re-derives `vat_amount` from the transaction's existing `vat_rate_id` when the amount changes on an income/expense row (reusing `resolveVatAmount()`, the same helper the main edit path uses) so it never goes stale relative to a corrected amount; transfers have no VAT so that step is skipped. Validated by `reconcileSchema` (`schema.ts`) — a deliberately reduced sibling of `transactionSchema`, not a reuse of it.
  - **Invoice** (2026-07, `invoice-button.tsx`/`invoice-modal.tsx`, `invoice_month` column) — a file-icon button, purely a personal bookkeeping cue: every month's invoices/receipts get collected into a dated folder for the accountant, and this just records *which* month's folder a given transaction's paperwork actually landed in (not necessarily the transaction's own date — e.g. a transaction dated 2 July might have its invoice filed under June). Opens a small modal with a single 1–12 number field; clearing it and saving unsets it. Lit (`text-accent`) with the month number as a small badge on the icon once set, same on/off visual language as Reconcile.
  - Both modal contexts share one dialog implementation (`transaction-form-dialog.tsx`, native `<dialog>`, unique field ids via `useId`), widened to `max-w-2xl` with fields laid out in a `sm:grid-cols-2` grid (2026-07, explicit user direction — the form had grown to ~9 fields stacked in a single `max-w-sm` column and was scrolling; Type/Description/the amount field/the summary box still span both columns, everything else pairs up two-per-row). Applies to the Edit dialog too, since it's the same component.

Also fixes a bug where fields carried over from the *previous* add (the dialog component was never unmounted between opens, so typed-in state simply persisted): `transaction-modal.tsx` now owns a `generation` counter that's bumped on every close/Add, and remounts `TransactionFormDialog` via `key={generation}` each time — a genuine fresh mount is what actually resets the entity/category comboboxes' own internal state, not just the parent's.
  - **Amount field, Net/Total toggle, and multi-VAT-rate lines** (2026-07, explicit user direction, several rounds) — Income/Expense's amount entry is a list of "lines" (`transaction_vat_lines` table — see Database section), almost always exactly one but occasionally more for a single invoice that mixes VAT rates (e.g. €1000 net @ 24% + €200 net @ 6%, one transaction, two lines). Each line is `[Amount input][VAT rate select]` on one row; a "+ Add VAT line" link appends another; a line gets a remove (×) button once there are 2+. A single Net/Total segmented toggle (matching the app's existing segmented-control pattern) sits next to the "Amount" section label, above the line rows — deliberately a single shared control rather than one per line (2026-07, explicit user direction: a real invoice is never going to mix "I know the net for this rate" with "I know the total for that rate," so one setting for the whole transaction is correct, not an oversight) — Income/Expense only, transfers have no VAT/lines at all and keep a single plain "Amount" field. In "Net" mode each line's typed number is net as-is; in "Total" mode it's treated as a total and the tax removed (`net = total / (1 + rate/100)`, `round2()`'d — exported from `src/lib/format.ts` and shared with `actions.ts`'s server-side VAT computation). The toggle is deliberately **not** a conversion step — switching modes leaves every line's typed number untouched and just reinterprets it, per explicit user direction ("the amount entered stays the same but the calculations change"). The summary box (Net/VAT amount/Total) sums across every line. The submitted `net`/`vat_rate_id` fields were replaced by a single hidden `lines` field holding a JSON-serialized array (`schema.ts`'s `linesField` parses and validates it) — FormData has no native array support, and this avoids hand-rolling bracket-notation field names. Server-side, `resolveFields()` (`actions.ts`) resolves each line's `vat_amount` the same way it always resolved the single one (never trusted from the client), and `fields.net`/`vat_amount` are always the *sum* across lines — every other reader of a transaction (Taxes, wallet balances, sorting/filtering, the list columns) keeps reading those two summed fields completely unchanged, oblivious to how many lines back them. `fields.vat_rate_id` is set only when there's exactly one line (null — "mixed rates" — otherwise). `writeLines()` replaces a transaction's lines wholesale (delete all, reinsert the new set) on both add and edit, which also naturally handles a type change (income/expense ↔ transfer) with no special-casing; `addTransaction` soft-deletes the just-inserted parent row if writing its lines fails, so a partial failure can't leave an orphaned transaction with no breakdown behind it. `reconcileTransaction()` only exposes one flat amount field (see Reconcile above), so a corrected total is distributed across a multi-line transaction's existing lines proportionally to their current share of the old net, keeping each line's own rate. `getActiveTransactions()`/`getWalletTransactionsWithBalance()` (`queries.ts`) both call a shared `attachVatLines()` that bulk-fetches lines for just the current page's rows (≤25) — this is the one piece the flattened `transactions_expanded` view can't carry (a one-to-many table can't flatten onto a single row) — populating each `Transaction.vatLines`, which only the edit dialog actually reads (to seed its line editor); `transaction-row.tsx` passes it straight through as `defaultValues.lines`.

Opening "+ Add" from here pre-fills the new category's type to match the transaction's current Income/Expense selection, still changeable. Transfer shows **From wallet** / **To wallet** selects instead (both required, and must differ — checked both client-side, for a quick inline error, and server-side/DB-side, since that's the actual source of truth).

Transfers have no VAT, so `vat_amount = 0` and `Total = net` for them.

If a future dialog puts padding directly on the `<dialog>`, clicking that padding will wrongly count as "outside" and close it — keep visual chrome on an inner wrapper.
  - **Transaction types**: `Income` and `Expense` each affect exactly one wallet (via `wallet_id`) — income adds `net + vat_amount` to its balance, expense subtracts it. `Transfer` moves money between two of the user's own wallets with no entity/VAT: `wallet_id` is the "from" side (subtracts `net`), `to_wallet_id` is the "to" side (adds `net`) — see Database section for the DB-level constraint enforcing this split, and the Wallets section for how a transfer shows up in both wallets' ledgers from one single row (not two linked rows — much simpler for edit/delete, since there's nothing to keep in sync).

A new field on the add/edit form, right under Date — it starts equal to whatever Date holds and keeps live-following it as Date changes, *until* the user edits Invoice date directly, at which point it becomes independent (tracked client-side via `invoiceDateTouched` state in `transaction-form-dialog.tsx`; re-derived on re-opening an existing transaction by checking whether its saved `invoice_date`/`date` already differ, so a deliberate past divergence doesn't silently get re-synced). For a transfer row: the **Entity** column shows the literal text "Transfer" instead of being blank (there's no real entity on a transfer, but an empty cell read as broken); the **Wallet** column shows the "from" wallet name with the "to" wallet on a second, smaller, muted line directly beneath it (`leading-none`, not a full extra table row) rather than one long "A → B" line, so a long wallet-pair doesn't force the whole row wider — deliberately kept tight enough not to visibly grow the row's height versus a normal single-line row.
  - **Balance view** (2026-07, `balance-view-control.tsx`) — replaced what used to be a separate `/wallets/[id]` ledger page: an always-visible "Balance view" button next to Add opens a wallet picker; choosing one sets `?balance=<walletId>` (merged into whatever filters are already on the URL, same as every other filter here — nothing else resets), and the *same* Transactions table narrows to just that wallet's transactions with the Wallet column swapped for a running-Balance one at the end (before actions) — everything else (Net/VAT/Total, Category/Entity/Description, row actions, every filter, tri-state sorting) keeps working exactly as it does in the normal view, since it's still `TransactionRow`/`TransactionTableHeader` rendering it, just with a `balanceMode` prop on. The idiomatic App Router pattern: bookmarkable/shareable, and the Server Component (`page.tsx`) reads them directly and re-queries, no client-side data duplication. `parseFilters()`/`parseSortParam()` in `page.tsx` validate every param (enum/UUID/date-format/number) before anything reaches `getActiveTransactions()` — an invalid value passed straight to `.eq`/`.or`/`.gte` on a typed column would error the whole query, not just match nothing, so invalid values are silently dropped instead (treated as "no filter").

The view flattens Entity/Wallet/Category names (and a computed `total`) onto each row specifically so sorting by those joined columns is just an ordinary `.order("entity_name", ...)` — PostgREST can't order a top-level query by an embedded resource's column directly, which is what made this worth a view rather than fighting that limitation. `page.tsx` re-fetches clamped to the last valid page if a filter change shrinks the result set out from under whatever page was requested.

  - **Historical import** (2026-07, `transactions/import/`, `exceljs` dependency) — a permanent "Import" button next to Add transaction, for bulk-loading the user's pre-existing accounting spreadsheet (a fixed personal format: `Date, Account, Income, Expense, Category, Counterparty, Description, VAT, Bacon` columns; header names are matched, not fixed positions, so a reordered sheet still works). Every row becomes a plain Income or Expense transaction — the user represents wallet-to-wallet transfers as an expense-row/income-row pair in their spreadsheet rather than one row, and explicitly chose (2026-07) to keep importing them that way rather than have the importer try to detect and merge pairs into a real Transfer, since a reliable signal to detect a pair doesn't exist in the data. A 3-step flow rather than the app's usual single-form `ModalShell` pattern, since there's no single "submit": **upload** (pick the `.xlsx`) → **preview** (`parseImportFile()`, a pure read/compute pass — nothing is written yet) → **commit** (`commitImport()`, the actual writes, given exactly the rows the preview already resolved, so the file is parsed only once). Amounts in the file are **totals**, not net — `net = total / (1 + rate/100)`, same formula/rounding as the transaction form's own Total mode. Missing Categories/Entities/Wallets referenced by the file are auto-created (explicit user direction); a missing/unrecognized VAT rate is **not** auto-created, it fails that row instead (VAT rates are few and deliberate, unlike the other three). One hardcoded alias, `WALLET_ALIASES` (`import-actions.ts`) — the user's spreadsheet calls their bank wallet "Alpha", matched to the existing "Alpha Bank" wallet rather than becoming a second, separate wallet (confirmed explicitly, not guessed — silently creating a duplicate would have split that account's real balance history across two rows). A blank Description falls back to that row's Category name; a "Bacon" column value of `0`/`-`/blank all mean "not invoiced yet" (`invoice_month` left null — 0 isn't a valid month, the DB constraint requires 1–12); a valid 1–12 Bacon value sets `invoice_month` directly, per-row (not "the current month" — confirmed explicitly, since the phrasing was ambiguous). The preview shows aggregate counts/totals and the full list of new wallets/categories/entities rather than a per-row table (825 rows isn't something a human can usefully eyeball row-by-row; the aggregate + an error list is the meaningful review surface) — any row that fails to resolve (bad date, no income/expense amount, unrecognized VAT rate) is listed with its reason and simply excluded from the commit rather than blocking the whole import. `commitImport()` writes in chunks of 200 rather than one all-or-nothing database transaction (no RPC/stored procedure) — an accepted tradeoff for a single-user one-off tool: every write is a normal insert, so a partial failure is fully recoverable by hand (soft-delete the partial batch, fix the data, re-run) rather than corrupting anything, and a failure message reports exactly how many rows made it in before it. `next.config.ts` raises the Server Action body-size limit to 10mb (default is 1mb) as headroom for a larger file than the one this was built against. **Known open item**: the Invoice-month button/feature (see Transactions section above) needs rework before the user's real historical file is actually uploaded through this — flagged by the user themselves, not yet scoped; ask before that upload happens, don't assume.

Same soft-delete CRUD pattern as Transactions (row click opens edit modal, trash icon soft-deletes). The Name and VAT number headers also each carry a `HeaderTextFilterPopover` (2026-07) bound to the same `q` param the toolbar search box already uses — another entry point onto the one search, not a second one.
- **Wallets** (`/wallets`) — Full CRUD list of where money is held (bank accounts, cash, etc.). Fields: name only. Row click opens the edit modal, same as every other table (2026-07 — previously the wallet name was a link to a dedicated per-wallet ledger page, so a separate pencil-icon button handled edit instead; now that the ledger has moved into Transactions' balance view — see that section — there's nowhere left for the name to link to, so it went back to plain text and the row reverted to the standard click-anywhere-to-edit pattern, dropping the one documented deviation from the shared table template this page used to have). Fine at this app's scale (a handful of wallets per user). `/wallets/[id]` still exists purely as a redirect into Transactions' balance view — see that section.
- **Taxes** (`/taxes`, redesigned 2026-07) — Read-only throughout; not a CRUD list, there's nothing to store, everything here is a live aggregate over transactions. `/taxes` itself is a small list of tax types (`tax-types.ts` — a **code-defined array, not a user-editable table** like Categories/VAT rates, since tax rules are law-defined rather than arbitrary user data; currently just VAT, add a new entry plus its own `/taxes/[slug]/page.tsx` when a second tax type is needed, since each one's computation is genuinely different — no single generic page could serve all of them) shown as cards. Each card shows **two figures**: "Payable this month" and "This month's net", both for the current period, both read directly off `getMonthlyVatLedger()` (`payableThisMonth`/`net` — see below; an earlier version of this card showed a lifetime `SUM(vat_amount)` across every transaction, which turned out to be actively misleading since it added input and output VAT together instead of netting them, then a version after that naively used "last month's raw net" as the payable figure before the rollover/installment ledger below existed — both since replaced). The card's content is deliberately VAT-specific (`taxes/page.tsx` has a comment flagging this) — a future second tax type won't necessarily fit this same two-figure shape, so it'll need its own branch rather than being forced into it.
  - **VAT rollover and installments** (2026-07, `taxes/queries.ts`, researched and confirmed against Greek VAT rules — see chat, and confirmed explicitly by the user): a period's raw net (`net` = output − input) isn't what's actually owed — `getMonthlyVatLedger()` walks every period **chronologically** (oldest first) threading two pieces of running state forward: a **credit carry** (a negative net, "πιστωτικό", rolls forward indefinitely to offset a later period's debit — this app doesn't model the real 5-year statute of limitations on unused credit, since a single ongoing company will use it long before that's relevant) and a **deferred installment** (a positive net after credit, "χρεωστικό", over €100 is split into two equal interest-free installments — half due with that period's own filing, half due the following month; €100 or under is paid in full immediately). The installment *option* is always treated as taken — there's no toggle, matching explicit user direction. Each period's row exposes `rolloverIn` (credit brought in), `payableThisMonth` (this period's own now-due share **plus** whatever the previous period deferred in), and `payableNextMonth` (the half **this** period's own filing pushes into the next one). Every month between the earliest VAT-bearing transaction and the later of (latest transaction) or (today) is walked, including zero-activity gap months — a deferred installment or carried credit still has to pass *through* a quiet month to reach the next one with real activity, so skipping gaps would silently break the chain. This ledger walk always runs over the **complete, unfiltered** history (`getMonthlyVatLedger()`) — later periods depend on every one before them, so it can never be computed over a filtered subset; `getMonthlyVatList()` filters/sorts/paginates the *already-computed* result for display, the same split (and the same reasoning) `getWalletTransactionsWithBalance()` already uses.
  - **`/taxes/vat`** — built on the shared table template (2026-07, explicit user direction — an earlier version hand-rolled a plain table with no per-column filters, breaking the app's standing "every table is the same" rule). All seven columns (Month, Income VAT, Expenses VAT, Net VAT, Roll over, Payable this month, Payable next month) are sortable and filterable — Month via `HeaderDateRangeFilterPopover` (comparing against the 1st of each period), the six money columns via `HeaderNumberRangeFilterPopover` — same shared filter popovers as every other table, and `TablePagination`. Deliberately **no search box and no Add button** (nothing free-text to search, nothing to create — every row is a computed month, not a record) and **no `max-w` cap** (seven columns need the room, same reasoning as Transactions). Defaults to **oldest-first** (an earlier version defaulted newest-first, reasoned as more useful for a tax report — the user overrode that to match the rest of the app's oldest-on-top convention). Sorting by anything other than Month will make Roll over/Payable this-next month stop reading as a coherent chain (each row depends on the one before it) — same accepted, documented tradeoff the old wallet ledger's running-balance column already had.
  - **VAT** (`/taxes/vat`) — a month-by-month breakdown table (newest month first): Output VAT (from income) | Input VAT (from expense) | Net, per Greek VAT law as explicitly described by the user: VAT collected on income is owed to the state each period, reduced by VAT already paid on expenses in that same period — a net *negative* month is a credit, not a payable amount (no automatic carry-forward to the next period is computed; shown as a plain signed number for the user to interpret). `getMonthlyVat()` (`taxes/queries.ts`) fetches every income/expense row and groups by `invoice_date`'s year+month in JS (`row.invoice_date.slice(0, 7)`) — not `date`, per the Invoice date rule above. Aggregated in JS rather than a SQL `GROUP BY`, the same pattern `getWalletBalances()` already uses: PostgREST doesn't support arbitrary grouping through the standard client, and this app's data volume makes an in-JS reduce entirely reasonable. Deliberately **doesn't use the shared table template** — it's a fixed, small set of calendar months, not a searchable/sortable/paginated list, so none of that machinery applies. Each month is a link into `/transactions?invoice_from=<first-of-month>&invoice_to=<last-of-month>` — "which transactions make up this month's VAT" reuses Transactions' existing table/filters entirely rather than a bespoke breakdown UI on the Taxes page itself.
- **Options** (`/options`) — App-level preferences, not identity settings (those moved to Account, see below): just an **Appearance** section (theme switcher, see Design system below) now — the VAT rates section that used to live here moved to Lists (below) in 2026-07, once a second reference list (Categories) showed up and the two needed a shared home.
- **Lists** (`/lists`, 2026-07) — a dedicated home for the app's growing set of reference/lookup lists (VAT rates today; Categories, new; more later — see Notes to myself for likely future ones). Reached from the main side nav, not tucked in Options, since these feed core data entry (the transaction form) rather than being device-level preferences. Grouped, collapsible master-detail layout, not a flat scrolling page — a well-established pattern for exactly this ("managing several small lookup lists") researched before building: a narrower nav (`lists-nav.tsx`) on the left groups lists by domain (currently one group, "Transactions", holding Categories and VAT rates — `lists-groups.ts` is the single data structure that defines every group, so adding a future list/group is editing that one array) with per-group collapse/expand (chevron, local state — the layout doesn't remount while navigating within `/lists/*`, so it persists for the session without needing URL/localStorage), and the selected list's full page renders on the right (`layout.tsx`). `/lists` itself (no list selected) is a plain "select a list" placeholder, not a redirect.
  - **VAT rates** (`/lists/vat-rates`) — unchanged behavior from its old home in Options, just moved wholesale (`vat-rate-*.ts(x)`, `revalidatePath` updated to match) and given the same first-class `ListPageHeader`-based page every other list in the app has, instead of being an `<h2>` subsection of a bigger settings page. Full CRUD (name, rate %), soft-delete, sortable Name/Rate headers, pagination, still no *toolbar* search box (short, rarely-changed list — that reasoning didn't change just because it moved) — but the Name header now carries a `HeaderTextFilterPopover` and the Rate header a `HeaderNumberRangeFilterPopover` (both 2026-07, bound to a `q` param and `rate_min`/`rate_max` respectively; `getVatRatesList()` gained `search`/`rateMin`/`rateMax` support to back them), so this page's only search/filter capability lives entirely in the header now rather than the toolbar.
  - **Categories** (`/lists/categories`, new, 2026-07) — full CRUD list (name, and a Type of Income or Expense) that populates the new Category field on the transaction form (see Transactions above). Same soft-delete pattern and file shape as Entities (`queries.ts`/`actions.ts`/`schema.ts`/`category-*.tsx`). Built on the shared table template with search *and* a Type column — a pill (reusing the existing `income`/`expense` semantic colors, same visual language as Transactions' own Type pill, via the new shared `TypePill` component — see Design system) with a `HeaderFilterPopover` to filter by it, so the Income/Expense split is visible and filterable right on the list, not just implied; the Name header also carries a `HeaderTextFilterPopover` (2026-07) bound to the same `q` param as the toolbar search box. Seeded with 5 starter rows (Fuel, Software as Expense; Grant, Commission, Project as Income) via direct SQL against the user's account, same one-off approach as seeding the display name earlier — nothing in the app does this seeding itself.
- **Account** (`/account`, 2026-07) — Identity settings, split out of what used to be one big Options page: **Profile** (a display name, new — see below), **Email**, and **Password** sections. `account/actions.ts` holds `updateEmail`/`updatePassword` (moved verbatim from the old `options/actions.ts`, now-deleted) plus the new `updateDisplayName`. Reached via the account-menu dropdown (see app shell below), not the side nav.
  - **Display name**: stored as `user_metadata.full_name` on the Supabase Auth user (`auth.users.raw_user_meta_data`) rather than a new `profiles` table — it's exposed on the JWT already, so `getCurrentUser()` (`src/lib/user.ts`) reads both email and name off a single `supabase.auth.getClaims()` call, same as `options/page.tsx` used to do for email alone. Falls back to the email's local part, then "Account", if unset. `updateDisplayName` deliberately skips `parseOrThrow`/zod (unlike most Server Actions in this app) — it's a plain `<form action={...}>` post with no client-side try/catch, matching its neighbors `updateEmail`/`updatePassword`; a thrown validation error there would hit Next.js's generic error boundary instead of the inline `?message=` banner all three already share, so it does the same manual check they do instead.

## Technical sections

- **Framework**: Next.js 16 (App Router, Turbopack), TypeScript, Tailwind CSS v4, ESLint. Scaffolded with `create-next-app`.
- **Data/validation libs**: `zod` validates every Server Action's input (see `schema.ts`/`vat-rate-schema.ts` next to each feature's `actions.ts`) — `parseOrThrow()` (`src/lib/validation.ts`) parses `formDataToRecord(formData)` (`src/lib/form-data.ts`) against the schema and throws a clean, user-facing message on the first issue, rather than trusting raw `FormData` casts. The transaction schema is a `discriminatedUnion` on `type`, mirroring the `transactions_type_fields_check` DB constraint at the app layer (income/expense need wallet+VAT rate, transfer needs two distinct wallets and nothing else) — validating both places is intentional: the app layer gives an immediate, friendly error; the DB constraint is the actual source of truth and can't be bypassed by an app-layer bug. `@tanstack/react-query` remains installed but unused.
- **Generated Supabase types**: `src/lib/supabase/database.types.ts` (generated via the Supabase MCP's `generate_typescript_types` — regenerate after schema-changing migrations, never hand-edit) is wired into both `createClient()` factories (`lib/supabase/client.ts`/`server.ts`) via `createBrowserClient<Database>()`/`createServerClient<Database>()`, so `.from()`/`.select()` calls are checked against the real schema (a typo'd table/column name is now a compile error). Existing hand-written domain types (`Transaction`, `Entity`, etc. in each feature's `queries.ts`) are unaffected — they're still what the rest of the app uses, since each query function's `.returns<T>()` call overrides the generic-inferred type anyway. **Gotcha**: the generator types every Postgres `numeric` column as TS `number`, but the Supabase JS client actually returns `numeric` values as strings at runtime — don't trust that file's `number` label for `net`/`vat_amount`/`rate`/`total`; keep doing `Number(...)` explicitly, same as before.
- **Auth/session plumbing**: `@supabase/ssr` with three client setups:
  - `src/lib/supabase/client.ts` — browser client
  - `src/lib/supabase/server.ts` — server client (Server Components/Actions)
  - `src/lib/supabase/proxy.ts` + `src/proxy.ts` — session-refresh and route protection. Next.js 16 renamed "middleware" to "proxy" (file `proxy.ts`, function `proxy`) — this app follows that convention. Auth check uses `supabase.auth.getClaims()` (not `getSession()`/`getUser()`), per current Supabase guidance.
  - Unauthenticated visitors are redirected to `/login` for any route except `/login`, `/signup`, and `/auth/*`.
- **Page structure**: `src/app/(app)/` is a route group holding the authenticated app shell — `layout.tsx` (top bar + side nav, see Main sections), `top-bar.tsx`/`side-nav.tsx`/`user-menu.tsx`/`notification-bell.tsx` (shell pieces), `page.tsx` (Home, `/`), `transactions/page.tsx` + `transactions/actions.ts` (`/transactions`), `account/page.tsx` (`/account`), `options/page.tsx` (`/options`). Route groups (parens in the folder name) don't add a URL segment. `/login`, `/signup`, and `/auth/*` live outside this group (no shell).
- **Database**: Supabase Postgres. Project `yolt-app`, ref `mzfxfweljbfvyqlhvmzr`, region `eu-west-1`, free tier, org "Angel-Pappas's Org" (`yowsydnsxjwcujsjwtji` — same org as the Exalted-Character-App project).
  - Table `public.transactions`: `id` (uuid pk), `user_id` (uuid, defaults to `auth.uid()`), `date` (date), `description` (text), `type` (text, `check (type in ('income','expense','transfer'))`), `net` (`numeric(12,2)`, `check (net >= 0)` — always a non-negative magnitude; `type` determines direction, the number itself never carries a sign), `entity_id` (uuid, nullable FK → `entities.id`, `on delete set null`), `category_id` (uuid, nullable FK → `categories.id`, `on delete set null`, 2026-07 — see Transactions section's Category bullet), `wallet_id` (uuid, **not null**, FK → `wallets.id` — the affected wallet for income/expense, the "from" wallet for a transfer), `to_wallet_id` (uuid, nullable, FK → `wallets.id` — only set for transfers, the "to" wallet), `vat_rate_id` (uuid, nullable FK → `vat_rates.id`, `on delete set null`), `vat_amount` (`numeric(12,2)`, not null default 0, `check (vat_amount >= 0)` — computed and stored server-side at save time, see Transactions section above for why), `created_at`, `is_deleted` (default false), `deleted_at` (nullable), `is_reconciled` (boolean, not null default false, 2026-07 — see Transactions section's Reconcile bullet), `invoice_month` (smallint, nullable, `check (invoice_month is null or invoice_month between 1 and 12)`, 2026-07 — see Transactions section's Invoice bullet), `invoice_date` (date, **not null**, 2026-07 — see Transactions section's Invoice date bullet; unrelated to `invoice_month` despite the similar name — two separate features that happen to both be about invoices). There is **no `amount` column anymore** — it was replaced by `net` + `vat_amount` (with `Total` derived, never stored). RLS enabled, all policies scoped to `auth.uid() = user_id`.
    - `wallet_id`'s and `to_wallet_id`'s FKs deliberately do **not** use `on delete set null` (unlike `entity_id`/`vat_rate_id`/`category_id`) — a hard-delete of a wallet that still has transaction history should fail loudly (Postgres default `NO ACTION`/restrict), not silently null out `wallet_id` (which would violate its `NOT NULL`) or silently turn a transfer into a broken half-transfer.
    - `transactions_type_fields_check` CHECK constraint ties `type` to which other fields are legal: `transfer` requires `to_wallet_id` set and different from `wallet_id`, plus `entity_id`/`vat_rate_id`/`category_id` all null and `vat_amount = 0`; `income`/`expense` require `to_wallet_id` null (`category_id` and, since 2026-07, `vat_rate_id` too are both left unconstrained — see `transaction_vat_lines` below for why `vat_rate_id` stopped being mandatory). This is enforced at the DB level, not just in the Server Action, so it can't be bypassed by a bug in the app layer. A category's own `type` (Income/Expense — see the `categories` table below) matching the transaction's `type` is *not* something a plain CHECK constraint can express (it can't reference another table) — that's validated in the app layer instead, in `resolveFields()` (`transactions/actions.ts`), by fetching the category's type server-side and rejecting a mismatch.
  - Table `public.transaction_vat_lines` (2026-07 — see Transactions section's "Amount field, Net/Total toggle, and multi-VAT-rate lines" bullet): `id`, `user_id` (default `auth.uid()`), `transaction_id` (uuid, FK → `transactions.id`, **`on delete cascade`** — unlike every other FK in this app, since a hard-delete of a transaction, however rare, should take its lines with it rather than fail or orphan them), `net` (`numeric(12,2)`, `check (net >= 0)`), `vat_rate_id` (uuid, nullable FK → `vat_rates.id`, `on delete set null`), `vat_amount` (`numeric(12,2)`, not null default 0, `check (vat_amount >= 0)` — computed and stored server-side the same way the parent's is), `position` (smallint, preserves entry order across edits), `created_at`. `transactions.net`/`vat_amount` remain the authoritative *summed* values across a transaction's lines (recomputed and stored server-side in `resolveFields()`/`writeLines()` on every add/edit) — deliberately denormalized so every other reader in the app (Taxes, wallet balances, sorting/filtering, list columns) keeps working against those two fields completely unchanged, oblivious to how many lines back them; `transaction_vat_lines` is purely the breakdown behind that sum, fetched only by the edit dialog. Same RLS pattern as every other table (`auth.uid() = user_id`), **except** it's the one table in the app with a real DELETE policy — a deliberate, documented exception: unlike every other table, lines aren't an independent record the user tracks (no soft-delete concept applies), they're wholesale replaced (`writeLines()`: delete all, reinsert the new set) every time a transaction's amount breakdown is saved.
  - Tables `public.entities`, `public.wallets`, `public.vat_rates`, `public.categories` (2026-07): same shape/conventions — `id`, `user_id` (default `auth.uid()`), `name`, plus one extra field each (`vat_number` for entities, `rate numeric(5,2)` for vat_rates, `type text check (type in ('income','expense'))` for categories; wallets has no extra field), `created_at`, `is_deleted`, `deleted_at`. Same RLS pattern (and same policy-naming convention — "Users can view/insert/update their own X") as transactions.
  - **Soft delete**: the app's "Delete" action does `update is_deleted = true, deleted_at = now()`, never a real `DELETE` — on **every** table (transactions, entities, wallets, vat_rates, categories). Filtering out `is_deleted = true` rows is done **only at the app-query level**, via a single shared helper per table (`getActiveTransactions()`, `getActiveEntities()`, `getActiveWallets()`, `getActiveVatRates()`, `getActiveCategories()`) — see the important RLS gotcha in Notes to myself: each table's SELECT RLS policy deliberately does *not* filter on `is_deleted`, because it can't without breaking the soft-delete UPDATE itself. There is **no DELETE policy** on any of these tables (dropped/never added deliberately) — the app has no way to hard-delete a row, by design. Only the user (owner) can ask for a permanent delete or a restore, done by directly running SQL via the Supabase MCP (`execute_sql`/`apply_migration`), not through any in-app feature.
  - Editing a `vat_rates` row's `name`/`rate` (or a `categories` row's `name`/`type`) only affects the dropdown/list label and future transactions — it never retroactively changes anything already stored on transactions that already used it, since that value was snapshotted at save time.
  - **`transactions_expanded`** (view, `public` schema): flattens `entity_name`, `category_name`, `wallet_name`, `to_wallet_name`, `vat_rate_name`/`vat_rate` (the percentage), and a computed `total` (`net + vat_amount`) onto each transaction row, so the Transactions page can filter/sort/paginate entirely via ordinary PostgREST query params against one resource — including sorting by Entity/Wallet, which PostgREST can't do directly against a top-level query when they're embedded/joined resources. Created `with (security_invoker = true)` — **essential**: without it, the view runs with its owner's privileges for RLS purposes and could leak every user's rows regardless of who's querying; with it, the view is evaluated as the calling user, so the existing RLS policies on the underlying tables apply exactly as if those tables were queried directly. Verified via the security advisor after every change (most recently the 2026-07 `invoice_date` addition) and it's found no issues each time. Granted `SELECT` to `authenticated`. `getActiveTransactions()` (`transactions/queries.ts`) queries this view, not the raw table, and maps its flat columns back into the app's normal nested `Transaction` shape so every consumer (`transaction-row.tsx`, the form dialog, etc.) is unaffected. Since the view lists its columns explicitly (not `select *`), adding a new column to `transactions` (e.g. `is_reconciled`/`invoice_month`, `category_id`) never flows through automatically — the view has to be recreated (`create or replace view ... with (security_invoker = true)`) to add it, same as each of those migrations did; don't assume a new transactions column is queryable via this view without checking.
  - **Indexes** (all partial, `where is_deleted = false`, added ahead of the ~750-row historical import): `transactions` on `(user_id, date)`, `type`, `wallet_id`, `to_wallet_id`, `entity_id`, `vat_rate_id`, `category_id` (2026-07), `(user_id, invoice_date)` (2026-07, backs the Taxes monthly VAT grouping and the `invoice_from`/`invoice_to` filter); a lighter `(user_id)` index on `entities`/`wallets`/`vat_rates`/`categories`. `transaction_vat_lines` (2026-07, not partial — no `is_deleted` concept there) on `transaction_id` (the real query pattern — fetching one transaction's lines) and `user_id`.
- **Hosting/deploy**: Vercel, connected to GitHub repo [Angel-Pappas/yolt-app](https://github.com/Angel-Pappas/yolt-app), auto-deploys on push to `main`. Env vars (`NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY`) are set directly in the Vercel dashboard (mirrors `.env.local`, which is gitignored; `.env.example` is committed as a template).

## Formatting conventions

Greek-style formatting is used everywhere the app displays dates/numbers: dates as `dd/mm/yyyy`, numbers with `.` as the thousands separator and `,` as the decimal separator, always rounded to exactly 2 decimals. Shared helpers live in `src/lib/format.ts` (`formatDate`, `formatAmount`, `computeTotal`, `formatMonthYear` — 2026-07, renders a "yyyy-mm" tax-period key as e.g. "July 2026" for the Taxes → VAT monthly breakdown; deliberately English month names via a fixed array rather than `toLocaleDateString`, since the rest of the app's UI text is English even though its date/number formatting is Greek-style) — use these instead of ad-hoc formatting/arithmetic whenever displaying a date or amount anywhere in the app. Native HTML form inputs (`<input type="date">`, `<input type="number">` in the add-transaction modal) are exempt — their `value` must stay ISO/period-decimal per the HTML spec, only their on-screen picker rendering is browser/OS-locale-dependent, and that isn't something the app controls.

## Design system

The app has a real visual identity now (2026-07), not default-Tailwind styling — this replaced a "crude"-looking first pass across every page.

- **Theme**: Light / Dark / Follow system, chosen in Options → Appearance (`theme-switcher.tsx`). Preference is a cookie (`yolt-theme`, `src/lib/theme.ts`), not per-account DB state — deliberate, since this is a device-level preference, not something that needs cross-device sync. `getThemePreference()` reads it server-side in the root layout (`src/app/layout.tsx`) to set `data-theme` on `<html>` **before first paint**, which is what avoids a flash of the wrong theme. When the preference is "system", `data-theme` is omitted entirely and a `@media (prefers-color-scheme: dark)` rule (scoped to `:root:not([data-theme])`) decides, purely via CSS — no JS involved for that case.
  - Reading `cookies()` in the root layout opts the **entire app** out of static rendering (a few routes — `/`, `/auth/auth-code-error` — were previously prerendered as static; now everything is server-rendered on demand). Accepted, deliberate tradeoff for correct SSR theming — not a regression to "fix."
  - The theme switcher applies instantly client-side (`document.documentElement.setAttribute("data-theme", ...)`) *and* persists via a Server Action (`options/theme-actions.ts`) that sets the cookie — so it feels instant without waiting for a round trip, but survives a refresh/new session too.
  - Dark is the original "muted ledger" palette (deep desaturated teal accent, near-navy ink); Light is a later, deliberately brighter iteration of the *same* hue families (more luminous/vivid accent + semantic colors, lighter ink/borders) — the user explicitly asked for light to read as "brighter," not just "not dark." If asked to adjust one theme's palette, don't assume the other should match — they were tuned independently on purpose.
- **Tokens**: defined as CSS custom properties in `src/app/globals.css` (`--ground`, `--surface`/`-raised`/`-header`/`-alt`, `--ink`/`-muted`/`-faint`, `--border`/`-strong`, `--accent`/`-ink`/`-soft`, `--income`/`--expense`/`--transfer`/`--success` each with a `-soft` background variant), then exposed as Tailwind utilities via a Tailwind v4 `@theme inline` block (`bg-canvas`, `bg-surface`, `bg-surface-header`, `bg-surface-alt`, `text-ink`, `text-ink-muted`, `border-edge`, `bg-accent`, `text-income`, `bg-income-soft`, `text-success`, `bg-success-soft`, etc.) — style components through these utility names, never hardcode a hex or a default Tailwind color (no more `text-red-600`/`bg-neutral-50`/etc. anywhere in the app). `income`/`expense`/`transfer` are semantic colors, deliberately a separate hue family from `accent` (the one used for primary actions/focus) — don't blur that distinction by reusing accent for a status meaning or vice versa. `--success` (2026-07) is a fourth, *also* separate hue — a status color (used only by the Reconcile button once a transaction is reconciled) kept deliberately distinct from `--income`'s green so "this is income" and "this is reconciled" never share a color that could read as one implying the other.
  - **2026-07 contrast pass**: `--ground` was pulled noticeably further from `--surface` in both themes (tables were blending into the page background — a card sitting on a near-identical ground read as un-separated text, not a distinct surface). `--surface-header` (a table header row tint) and `--surface-alt` (zebra-row tint) are new tokens added for the same reason. `--border-strong` was darkened slightly alongside this pass since it's now also used for the header's bottom border.
- **Typography**: two font roles, both via `@theme`'s `--font-display` / `--font-sans` — `font-display` (Cambria/Georgia/serif stack) for page titles and dialog headings, `font-sans` (system-ui stack) for everything else including all table data. No embedded/Google-fonts webfont (the previous unused Geist Google Font imports in `layout.tsx` were removed — they were loaded but never actually rendered, since `body`'s font-family was hardcoded to Arial, overriding them). Every numeric table/summary column gets Tailwind's `tabular-nums` so figures actually line up.
- **Component patterns** (apply these to any new page/feature rather than inventing new ones):
  - Card: `rounded-xl border border-edge bg-surface shadow-[var(--shadow-card)]` (a page's list/table lives in one of these, not floating unstyled).
  - **Page container width (2026-07)**: every page's top-level div is `flex w-full ... flex-1 flex-col gap-6 p-6`, **never** `mx-auto` — centering made each page float in the middle of the space next to the sidebar, and since pages used different `max-w` values, they landed at different distances from the sidebar (the main source of the app feeling "not uniform" from page to page). Left-aligned instead: every page starts at the exact same x-position, right after the sidebar. Width itself still varies by content, deliberately — Transactions (9–10 columns, including its balance view) has **no `max-w` cap at all**, so their table gets to use however much room the window actually gives it before the table's own `overflow-x-auto` has to kick in ("no scrollbar unless the screen is actually full," explicit user direction); Entities/Wallets/Taxes/Home keep `max-w-5xl`, Options/Account keep `max-w-2xl` — content-appropriate widths, not stretched to match Transactions, since forcing a 3-column table into a wide container would strand its Add button far from the actual table (the thing to avoid). If a future page needs a different width, follow this same reasoning (cap only if the content doesn't need the room) rather than copying one page's number by default.
  - **The shared table template** (`src/components/table/`, 2026-07): every list page in the app (Transactions — including its balance view, Entities, Wallets, VAT rates, Categories, `/taxes/vat`, and anything added later) is built from the same small set of shared pieces instead of each page hand-rolling its own — a direct, explicit user request: "all tables should be the same," with a future style/behavior tweak meant to happen in one file and show up everywhere. This applies even to read-only report-style tables, not just CRUD lists — `/taxes/vat` originally hand-rolled its own plain table and was rebuilt onto this template once that was pointed out, same as every other list in the app. The pieces:

- A thrown error inside a `startTransition(async () => { await action(...) })` callback does **not** reach an Error Boundary and is *not* automatically shown anywhere — it becomes a silent unhandled promise rejection unless caught explicitly. Every modal form dialog and delete button wraps its action call in try/catch (dialogs: `useState<string|null>` displayed near the buttons; delete buttons: `alert()`) — keep this pattern for any new Server-Action-calling client component, don't assume a thrown error will surface on its own.
- Money is stored as Postgres `numeric(12,2)`, which is exact fixed-point decimal — no floating-point rounding risk. Don't "fix" this into cents/bigint, it's already correct as-is.
- Derived/aggregate values (wallet balances, the VAT figures on the Taxes page, `Total` on transactions) are **never stored** — always computed live from their inputs at read time. This was a deliberate choice over a maintained running counter: no increment/decrement logic to get right on every create/edit/delete/restore, and it can't ever drift. Keep following this for any future aggregate (don't add a stored counter without discussing it first).
- A `<dialog>` containing a `<form>` must never be nested *inside* another `<form>` — nested `<form>` elements are invalid HTML and get mangled by the browser's parser during SSR (the inner `<form>` tag is silently dropped since a form element is already open), causing a hydration mismatch. Where a modal-within-a-modal is needed (e.g. the "+ Add" entity button on the transaction form, which opens the entity-add dialog), render the inner `<dialog>` as a **sibling** of the outer `<form>` — a child of the outer `<dialog>`, not of the inner `<form>` — and trigger it via a ref from a plain button inside the outer form. - **Postgres RLS gotcha (learned the hard way, don't repeat it on future tables)**: an `UPDATE`'s resulting row must satisfy the table's `SELECT` policy too, not just the `UPDATE` policy's own `WITH CHECK` — this is enforced unconditionally, regardless of what the `UPDATE` policy's check says. So a SELECT policy that filters out `is_deleted = true` makes it *structurally impossible* for a user to ever set `is_deleted = true` on their own row (the resulting row would be invisible to their own write, which Postgres blocks outright — this produces the exact error `new row violates row-level security policy`). The fix: for any table with a soft-delete column, the SELECT RLS policy must stay unrestricted on that column (just `auth.uid() = user_id`, no `is_deleted` condition) — do the "hide deleted rows" filtering only in a shared app-level query helper. `categories` (2026-07) already followed this correctly; keep doing so for any future table that gets soft-delete.
