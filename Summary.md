# Summary

This file exists so a fresh session can get up to speed on this app quickly. Keep it current: update the relevant section whenever a major feature is added, removed, or changed.

## General description

Yolt-App is a personal financial transactions tracker, built for a single user (the app owner only — no multi-tenant/sharing features are planned). It supports logging typed transactions (Income, Expense, or Transfer between wallets — date, entity, wallet(s), description, net amount, VAT) behind email/password authentication, plus supporting lists of Entities, Wallets, and VAT rates, and a Taxes overview. Planned future features include financial projections, automatic generation of recurring transactions, richer tax handling (multiple tax types, input/output VAT netting), a real notifications system (currently a placeholder — see Options/Account section below), plus whatever else comes up as the app grows.

## Main sections of the app

- **Authentication** — Signup, login, and logout. Email/password via Supabase Auth, including the email-confirmation flow. Pages: `/login`, `/signup`, `/auth/confirm` (confirmation link handler), `/auth/auth-code-error` (expired/invalid link fallback).
- **Home** (`/`) — Minimal landing page after login. Placeholder content for now.
- **Transactions** (`/transactions`) — Full CRUD on transactions (date, type, entity, wallet(s), description, net, VAT rate), all backed by RLS-scoped Server Actions (`actions.ts`: `addTransaction`, `updateTransaction`, `deleteTransaction`).
  - **Add**: a text "Add" button opens a modal (`transaction-modal.tsx`).
  - **Edit**: no button — clicking anywhere on a transaction's row (`transaction-row.tsx`, a client component) opens the same edit modal, pre-filled.
  - **Delete**: a trash-icon button per row (`delete-transaction-button.tsx`, `src/components/icons.tsx` for the icon — inline SVG, no icon library dependency); its click handler (and the dialog's) calls `stopPropagation()` so it doesn't also trigger the row's edit-open behavior. Delete is a **soft delete** — see the Database section below. Nothing is ever hard-deleted through the app.
  - **Reconcile** (2026-07, `reconcile-button.tsx`/`reconcile-modal.tsx`, `is_reconciled` column) — a check-circle button that opens a **reduced** edit modal: only Date, Amount, and Wallet(s) are editable (no description/entity/VAT rate). Built ahead of automatic recurring-transaction generation (planned, not built yet — see Notes to myself): a future transaction gets auto-created with a projected date/amount/wallet, and once it actually happens the user opens Reconcile to correct whichever of those three drifted from the projection, then the row is marked reconciled regardless of whether anything actually changed — reconciling is itself the record that the user checked it, not just a change-tracking flag. The button lights up (`text-accent`) once `is_reconciled` is true. `reconcileTransaction()` (`actions.ts`) re-derives `vat_amount` from the transaction's existing `vat_rate_id` when the amount changes on an income/expense row (reusing `resolveVatAmount()`, the same helper the main edit path uses) so it never goes stale relative to a corrected amount; transfers have no VAT so that step is skipped. Validated by `reconcileSchema` (`schema.ts`) — a deliberately reduced sibling of `transactionSchema`, not a reuse of it.
  - **Invoice** (2026-07, `invoice-button.tsx`/`invoice-modal.tsx`, `invoice_month` column) — a file-icon button, purely a personal bookkeeping cue: every month's invoices/receipts get collected into a dated folder for the accountant, and this just records *which* month's folder a given transaction's paperwork actually landed in (not necessarily the transaction's own date — e.g. a transaction dated 2 July might have its invoice filed under June). Opens a small modal with a single 1–12 number field; clearing it and saving unsets it. Lit (`text-accent`) with the month number as a small badge on the icon once set, same on/off visual language as Reconcile.
  - Both modal contexts share one dialog implementation (`transaction-form-dialog.tsx`, native `<dialog>`, unique field ids via `useId`).
  - **Type** (`income` / `expense` / `transfer`) is picked first in the form and drives which other fields render — see the "Transaction types" subsection below.
  - **Entity** field (Income/Expense only) is a client-side-filtered search-and-select (`entity-combobox.tsx`) over the already-loaded entity list, with a hidden `entity_id` input and a "+ Add" button that opens the entity-add dialog (`entities/entity-form-dialog.tsx`) as a **sibling** of the transaction `<form>`, not nested inside it — nesting a `<dialog><form>` inside another `<form>` is invalid HTML and breaks under SSR. No inline creation from the combobox itself; the user must pick the newly-created entity from the search afterward.
  - **Wallet** and **VAT** (Income/Expense) are plain `<select>` dropdowns (populated from active wallets / active VAT rates). Transfer shows **From wallet** / **To wallet** selects instead (both required, and must differ — checked both client-side, for a quick inline error, and server-side/DB-side, since that's the actual source of truth).
  - **VAT Amount** and **Total** are calculated, read-only (live-updated client-side as Net/VAT change for display, but never trusted from the client): `vat_amount` is computed and stored server-side in the Server Action from `net × the selected rate's current percentage` at save time (so editing a VAT rate's percentage later doesn't retroactively change past transactions' stored VAT amount). `Total` (`net + vat_amount`) is never stored at all — always computed on read via `computeTotal()` in `src/lib/format.ts`. Transfers have no VAT, so `vat_amount = 0` and `Total = net` for them.
  - **Modals close on Esc (native, free) and on clicking the backdrop.** The backdrop-click handler checks that the click landed exactly on the `<dialog>` element itself (`e.target === e.currentTarget`) — which only happens for the backdrop, *because* each dialog's padding/border/background live on the inner `<form>`, not the `<dialog>`. If a future dialog puts padding directly on the `<dialog>`, clicking that padding will wrongly count as "outside" and close it — keep visual chrome on an inner wrapper.
  - **Transaction types**: `Income` and `Expense` each affect exactly one wallet (via `wallet_id`) — income adds `net + vat_amount` to its balance, expense subtracts it. `Transfer` moves money between two of the user's own wallets with no entity/VAT: `wallet_id` is the "from" side (subtracts `net`), `to_wallet_id` is the "to" side (adds `net`) — see Database section for the DB-level constraint enforcing this split, and the Wallets section for how a transfer shows up in both wallets' ledgers from one single row (not two linked rows — much simpler for edit/delete, since there's nothing to keep in sync).
  - **Filtering, sorting, pagination**: all state lives in the URL's search params (`q`, `type`, `entity`, `wallet`, `vat`, `from`, `to`, `sort`, `dir`, `page`) via `useListParams()`, part of the **shared table template** (`src/components/table/` — see Design system below) used by every list page in the app, not just this one. The idiomatic App Router pattern: bookmarkable/shareable, and the Server Component (`page.tsx`) reads them directly and re-queries, no client-side data duplication. `parseFilters()`/`parseSort()` in `page.tsx` validate every param (enum/UUID/date-format) before anything reaches `getActiveTransactions()` — an invalid value passed straight to `.eq`/`.or` on a uuid column would error the whole query, not just match nothing, so invalid values are silently dropped instead (treated as "no filter").
    - **`transaction-filters-bar.tsx`**: composes the shared `SearchBox` (description only — Entity/Wallet/VAT have their own per-column filter instead, see below) and `DateRangeFilter` (quick presets: this month/last month/this year/all time) from the shared table template, plus a `ClearFiltersLink`.
    - **`transaction-table-header.tsx`**: every column header is a `SortableHeaderCell` (▲/▼ indicator on the active one; click again to flip direction) — Type/Entity/Wallet/VAT headers additionally carry their own `HeaderFilterPopover` in the header cell. Wallet filtering matches either side of a transfer (`wallet_id` or `to_wallet_id`), same as the wallet ledger.
    - **Sorting and pagination happen at the database level** via the `transactions_expanded` view (see Database section) — `getActiveTransactions()` queries that view with real `.order()`/`.range()` and an exact count, instead of fetching everything and sorting/paginating in JS. The view flattens Entity/Wallet/VAT-rate names (and a computed `total`) onto each row specifically so sorting by those joined columns is just an ordinary `.order("entity_name", ...)` — PostgREST can't order a top-level query by an embedded resource's column directly, which is what made this worth a view rather than fighting that limitation. `page.tsx` re-fetches clamped to the last valid page if a filter change shrinks the result set out from under whatever page was requested.
    - Changing any filter or sort resets pagination back to page 1 (`setFilterParams` in `useListParams()` always clears `page`); only `setPage` (Previous/Next) touches the page param alone.
    - Deliberately **not built**: a totals/summary bar (tried, then explicitly removed — not useful for this app) and CSV export (out of scope for now).
- **Entities** (`/entities`) — Full CRUD list of the counterparties transactions are made with (suppliers, customers, the state, etc.). Fields: name (required), VAT number (optional). Same soft-delete CRUD pattern as Transactions (row click opens edit modal, trash icon soft-deletes). Built on the shared table template: search (matches name OR VAT number, via `getEntitiesList()`), sortable Name/VAT number headers, pagination — all DB-level, same shape as Transactions.
- **Wallets** (`/wallets`) — Full CRUD list of where money is held (bank accounts, cash, etc.). Fields: name only. No stored balance column — the list page shows each wallet's **current balance computed live** (`getWalletBalances()` in `wallets/queries.ts`), so it can never drift out of sync as transactions are added/edited/deleted/restored. Since the wallet name is a link (to the ledger view below) rather than the whole row, edit uses a pencil-icon button (`wallet-row.tsx`) instead of the row-click pattern used elsewhere — the one table that doesn't use click-anywhere-to-edit, since the row already has a different, more prominent click target (the name link). Built on the shared table template: search (name), sortable Name/Balance headers, pagination — via `getWalletsList()`. Unlike Entities/VAT rates, **sorting and pagination happen in JS, not at the database level**, for every sort key (not just Balance) — Balance isn't a stored column, so there's no DB column to `.order()` by when sorted on it; rather than switch code paths depending on which column is active, it always fetches all matching wallets, computes balances, sorts, and paginates in memory. Fine at this app's scale (a handful of wallets per user).
  - **Ledger view** (`/wallets/[id]`) — a per-wallet transaction timeline with a running balance column, computed by walking that wallet's transactions in chronological (oldest-first) order (`getWalletLedger()`), matching the rest of the app. A transfer shows up in **both** wallets' ledgers (it's one row, matched by `wallet_id = this wallet OR to_wallet_id = this wallet`), signed from each wallet's own point of view and labeled the same "X → Y" way as the Transactions page (the actual from/to wallet names, not "Transfer" + an arrow). Built on the full shared table template, same capabilities as Transactions (search, date range, sortable/filterable Type column, pagination), and actually reuses `ListPageHeader` directly (`addButton` omitted — this page has none, transactions are added from the Transactions page — so the search box lands where the Add button would be, per explicit user direction) rather than a lookalike copy. `getWalletLedger()` always computes the running balance over the wallet's *complete* history first (querying `transactions_expanded` scoped via `.or(wallet_id.eq / to_wallet_id.eq)`, the same idiom as Transactions' Wallet filter), and only *then* applies search/type/date-range/sort/pagination to that already-computed list for display — so the per-row balance stays historically accurate no matter what's filtered out of view. Every column is sortable, including Amount/Balance — sorting by anything other than Date will make the running-balance column stop reading top-to-bottom as a sensible tally (each number is still correct, just no longer telling a clean story out of date order); accepted as a rare, low-stakes edge case rather than special-cased away, per explicit user direction. **No headline "current balance" figure on this page** (removed 2026-07, explicit user direction — the wallet's balance already sits in the Wallets list, and duplicating it here as a large number next to the title was unwanted); the per-row running-balance column already surfaces the same information where it's relevant.
- **Taxes** (`/taxes`) — Read-only. Currently shows a single live-computed total (`SUM(vat_amount)` across active transactions, via `getTotalVat()`). Not a CRUD list — there's nothing to store, it's purely an aggregate over transactions. More tax types (and more intricacies — input/output VAT netting, filing periods, etc.) are expected here later.
- **Options** (`/options`) — App-level preferences, not identity settings (those moved to Account, see below): an **Appearance** section (theme switcher, see Design system below) first, then a **VAT rates** section: full CRUD list (name, rate %) that populates the VAT dropdown on the transaction form. Same soft-delete pattern; files `vat-rate-*.ts(x)`. Built on the shared table template minus the search box (`getVatRatesList()`) — sortable Name/Rate headers and pagination, but no `SearchBox`: this list is short and rarely changes (a handful of rates, set once and barely touched again), so a search box would be clutter. Still shares the same header/row look and sort mechanism as every other table.
- **Account** (`/account`, 2026-07) — Identity settings, split out of what used to be one big Options page: **Profile** (a display name, new — see below), **Email**, and **Password** sections. `account/actions.ts` holds `updateEmail`/`updatePassword` (moved verbatim from the old `options/actions.ts`, now-deleted) plus the new `updateDisplayName`. Reached via the account-menu dropdown (see app shell below), not the side nav.
  - **Display name**: stored as `user_metadata.full_name` on the Supabase Auth user (`auth.users.raw_user_meta_data`) rather than a new `profiles` table — it's exposed on the JWT already, so `getCurrentUser()` (`src/lib/user.ts`) reads both email and name off a single `supabase.auth.getClaims()` call, same as `options/page.tsx` used to do for email alone. Falls back to the email's local part, then "Account", if unset. `updateDisplayName` deliberately skips `parseOrThrow`/zod (unlike most Server Actions in this app) — it's a plain `<form action={...}>` post with no client-side try/catch, matching its neighbors `updateEmail`/`updatePassword`; a thrown validation error there would hit Next.js's generic error boundary instead of the inline `?message=` banner all three already share, so it does the same manual check they do instead.
- All pages above share a common app shell (`src/app/(app)/layout.tsx`, a route group), redesigned 2026-07 into a "classic application" layout: a full-width **top bar** (`top-bar.tsx` — deliberately has no `max-w`/`mx-auto` wrapper, unlike every page's own content, so it spans the true screen edges regardless of content width) showing "Yolt-App" far left and a **notification bell** + **account menu** far right, plus a fixed-width **side nav** (`side-nav.tsx`, replaces the old horizontal `nav-links.tsx` — Home / Transactions / Entities / Wallets / Taxes, same active-link matching logic as before) running down the left of the content area. The account menu (`user-menu.tsx`, kebab-icon trigger, reuses the same outside-click-to-close popover mechanics as `HeaderFilterPopover`) shows the display name + email, then Account / Options / Log out. The notification bell (`notification-bell.tsx`) is a placeholder for a real feature later — always shows one static "Notifications coming soon" entry with a permanent unread dot, not wired to any data yet.

## Technical sections

- **Framework**: Next.js 16 (App Router, Turbopack), TypeScript, Tailwind CSS v4, ESLint. Scaffolded with `create-next-app`.
- **Data/validation libs**: `zod` validates every Server Action's input (see `schema.ts`/`vat-rate-schema.ts` next to each feature's `actions.ts`) — `parseOrThrow()` (`src/lib/validation.ts`) parses `formDataToRecord(formData)` (`src/lib/form-data.ts`) against the schema and throws a clean, user-facing message on the first issue, rather than trusting raw `FormData` casts. The transaction schema is a `discriminatedUnion` on `type`, mirroring the `transactions_type_fields_check` DB constraint at the app layer (income/expense need wallet+VAT rate, transfer needs two distinct wallets and nothing else) — validating both places is intentional: the app layer gives an immediate, friendly error; the DB constraint is the actual source of truth and can't be bypassed by an app-layer bug. `@tanstack/react-query` remains installed but unused.
- **Generated Supabase types**: `src/lib/supabase/database.types.ts` (generated via the Supabase MCP's `generate_typescript_types` — regenerate after schema-changing migrations, never hand-edit) is wired into both `createClient()` factories (`lib/supabase/client.ts`/`server.ts`) via `createBrowserClient<Database>()`/`createServerClient<Database>()`, so `.from()`/`.select()` calls are checked against the real schema (a typo'd table/column name is now a compile error). Existing hand-written domain types (`Transaction`, `Entity`, etc. in each feature's `queries.ts`) are unaffected — they're still what the rest of the app uses, since each query function's `.returns<T>()` call overrides the generic-inferred type anyway. **Gotcha**: the generator types every Postgres `numeric` column as TS `number`, but the Supabase JS client actually returns `numeric` values as strings at runtime — don't trust that file's `number` label for `net`/`vat_amount`/`rate`/`total`; keep doing `Number(...)` explicitly, same as before.
- **Auth/session plumbing**: `@supabase/ssr` with three client setups:
  - `src/lib/supabase/client.ts` — browser client
  - `src/lib/supabase/server.ts` — server client (Server Components/Actions)
  - `src/lib/supabase/proxy.ts` + `src/proxy.ts` — session-refresh and route protection. Next.js 16 renamed "middleware" to "proxy" (file `proxy.ts`, function `proxy`) — this app follows that convention. Auth check uses `supabase.auth.getClaims()` (not `getSession()`/`getUser()`), per current Supabase guidance.
  - Unauthenticated visitors are redirected to `/login` for any route except `/login`, `/signup`, and `/auth/*`.
- **Page structure**: `src/app/(app)/` is a route group holding the authenticated app shell — `layout.tsx` (top bar + side nav, see Main sections), `top-bar.tsx`/`side-nav.tsx`/`user-menu.tsx`/`notification-bell.tsx` (shell pieces), `page.tsx` (Home, `/`), `transactions/page.tsx` + `transactions/actions.ts` (`/transactions`), `account/page.tsx` (`/account`), `options/page.tsx` (`/options`). Route groups (parens in the folder name) don't add a URL segment. `/login`, `/signup`, and `/auth/*` live outside this group (no shell).
- **Database**: Supabase Postgres. Project `yolt-app`, ref `mzfxfweljbfvyqlhvmzr`, region `eu-west-1`, free tier, org "Angel-Pappas's Org" (`yowsydnsxjwcujsjwtji` — same org as the Exalted-Character-App project).
  - Table `public.transactions`: `id` (uuid pk), `user_id` (uuid, defaults to `auth.uid()`), `date` (date), `description` (text), `type` (text, `check (type in ('income','expense','transfer'))`), `net` (`numeric(12,2)`, `check (net >= 0)` — always a non-negative magnitude; `type` determines direction, the number itself never carries a sign), `entity_id` (uuid, nullable FK → `entities.id`, `on delete set null`), `wallet_id` (uuid, **not null**, FK → `wallets.id` — the affected wallet for income/expense, the "from" wallet for a transfer), `to_wallet_id` (uuid, nullable, FK → `wallets.id` — only set for transfers, the "to" wallet), `vat_rate_id` (uuid, nullable FK → `vat_rates.id`, `on delete set null`), `vat_amount` (`numeric(12,2)`, not null default 0, `check (vat_amount >= 0)` — computed and stored server-side at save time, see Transactions section above for why), `created_at`, `is_deleted` (default false), `deleted_at` (nullable), `is_reconciled` (boolean, not null default false, 2026-07 — see Transactions section's Reconcile bullet), `invoice_month` (smallint, nullable, `check (invoice_month is null or invoice_month between 1 and 12)`, 2026-07 — see Transactions section's Invoice bullet). There is **no `amount` column anymore** — it was replaced by `net` + `vat_amount` (with `Total` derived, never stored). RLS enabled, all policies scoped to `auth.uid() = user_id`.
    - `wallet_id`'s and `to_wallet_id`'s FKs deliberately do **not** use `on delete set null` (unlike `entity_id`/`vat_rate_id`) — a hard-delete of a wallet that still has transaction history should fail loudly (Postgres default `NO ACTION`/restrict), not silently null out `wallet_id` (which would violate its `NOT NULL`) or silently turn a transfer into a broken half-transfer.
    - `transactions_type_fields_check` CHECK constraint ties `type` to which other fields are legal: `transfer` requires `to_wallet_id` set and different from `wallet_id`, plus `entity_id`/`vat_rate_id` both null and `vat_amount = 0`; `income`/`expense` require `to_wallet_id` null and `vat_rate_id` set. This is enforced at the DB level, not just in the Server Action, so it can't be bypassed by a bug in the app layer.
  - Tables `public.entities`, `public.wallets`, `public.vat_rates`: same shape/conventions — `id`, `user_id` (default `auth.uid()`), `name`, plus one extra field each (`vat_number` for entities, `rate numeric(5,2)` for vat_rates; wallets has no extra field), `created_at`, `is_deleted`, `deleted_at`. Same RLS pattern as transactions.
  - **Soft delete**: the app's "Delete" action does `update is_deleted = true, deleted_at = now()`, never a real `DELETE` — on **every** table (transactions, entities, wallets, vat_rates). Filtering out `is_deleted = true` rows is done **only at the app-query level**, via a single shared helper per table (`getActiveTransactions()`, `getActiveEntities()`, `getActiveWallets()`, `getActiveVatRates()`) — see the important RLS gotcha in Notes to myself: each table's SELECT RLS policy deliberately does *not* filter on `is_deleted`, because it can't without breaking the soft-delete UPDATE itself. There is **no DELETE policy** on any of these tables (dropped/never added deliberately) — the app has no way to hard-delete a row, by design. Only the user (owner) can ask for a permanent delete or a restore, done by directly running SQL via the Supabase MCP (`execute_sql`/`apply_migration`), not through any in-app feature.
  - Editing a `vat_rates` row's `name`/`rate` only affects the dropdown label and future transactions — it never retroactively changes `vat_amount` on transactions that already used that rate, since that value was snapshotted at save time.
  - **`transactions_expanded`** (view, `public` schema): flattens `entity_name`, `wallet_name`, `to_wallet_name`, `vat_rate_name`/`vat_rate` (the percentage), and a computed `total` (`net + vat_amount`) onto each transaction row, so the Transactions page can filter/sort/paginate entirely via ordinary PostgREST query params against one resource — including sorting by Entity/Wallet, which PostgREST can't do directly against a top-level query when they're embedded/joined resources. Created `with (security_invoker = true)` — **essential**: without it, the view runs with its owner's privileges for RLS purposes and could leak every user's rows regardless of who's querying; with it, the view is evaluated as the calling user, so the existing RLS policies on the underlying tables apply exactly as if those tables were queried directly. Verified via `pg_class.reloptions` after creation and the security advisor found no issues. Granted `SELECT` to `authenticated`. `getActiveTransactions()` (`transactions/queries.ts`) queries this view, not the raw table, and maps its flat columns back into the app's normal nested `Transaction` shape so every consumer (`transaction-row.tsx`, the form dialog, etc.) is unaffected. Since the view lists its columns explicitly (not `select *`), adding a new column to `transactions` (e.g. `is_reconciled`/`invoice_month`, 2026-07) never flows through automatically — the view has to be recreated (`create or replace view ... with (security_invoker = true)`) to add it, same as that migration did; don't assume a new transactions column is queryable via this view without checking.
  - **Indexes** (all partial, `where is_deleted = false`, added ahead of the ~750-row historical import): `transactions` on `(user_id, date)`, `type`, `wallet_id`, `to_wallet_id`, `entity_id`, `vat_rate_id`; a lighter `(user_id)` index on `entities`/`wallets`/`vat_rates`.
- **Hosting/deploy**: Vercel, connected to GitHub repo [Angel-Pappas/yolt-app](https://github.com/Angel-Pappas/yolt-app), auto-deploys on push to `main`. Env vars (`NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY`) are set directly in the Vercel dashboard (mirrors `.env.local`, which is gitignored; `.env.example` is committed as a template).

## Formatting conventions

Greek-style formatting is used everywhere the app displays dates/numbers: dates as `dd/mm/yyyy`, numbers with `.` as the thousands separator and `,` as the decimal separator, always rounded to exactly 2 decimals. Shared helpers live in `src/lib/format.ts` (`formatDate`, `formatAmount`, `computeTotal`) — use these instead of ad-hoc formatting/arithmetic whenever displaying a date or amount anywhere in the app. Native HTML form inputs (`<input type="date">`, `<input type="number">` in the add-transaction modal) are exempt — their `value` must stay ISO/period-decimal per the HTML spec, only their on-screen picker rendering is browser/OS-locale-dependent, and that isn't something the app controls.

## Design system

The app has a real visual identity now (2026-07), not default-Tailwind styling — this replaced a "crude"-looking first pass across every page.

- **Theme**: Light / Dark / Follow system, chosen in Options → Appearance (`theme-switcher.tsx`). Preference is a cookie (`yolt-theme`, `src/lib/theme.ts`), not per-account DB state — deliberate, since this is a device-level preference, not something that needs cross-device sync. `getThemePreference()` reads it server-side in the root layout (`src/app/layout.tsx`) to set `data-theme` on `<html>` **before first paint**, which is what avoids a flash of the wrong theme. When the preference is "system", `data-theme` is omitted entirely and a `@media (prefers-color-scheme: dark)` rule (scoped to `:root:not([data-theme])`) decides, purely via CSS — no JS involved for that case.
  - Reading `cookies()` in the root layout opts the **entire app** out of static rendering (a few routes — `/`, `/auth/auth-code-error` — were previously prerendered as static; now everything is server-rendered on demand). Accepted, deliberate tradeoff for correct SSR theming — not a regression to "fix."
  - The theme switcher applies instantly client-side (`document.documentElement.setAttribute("data-theme", ...)`) *and* persists via a Server Action (`options/theme-actions.ts`) that sets the cookie — so it feels instant without waiting for a round trip, but survives a refresh/new session too.
  - Dark is the original "muted ledger" palette (deep desaturated teal accent, near-navy ink); Light is a later, deliberately brighter iteration of the *same* hue families (more luminous/vivid accent + semantic colors, lighter ink/borders) — the user explicitly asked for light to read as "brighter," not just "not dark." If asked to adjust one theme's palette, don't assume the other should match — they were tuned independently on purpose.
- **Tokens**: defined as CSS custom properties in `src/app/globals.css` (`--ground`, `--surface`/`-raised`/`-header`/`-alt`, `--ink`/`-muted`/`-faint`, `--border`/`-strong`, `--accent`/`-ink`/`-soft`, `--income`/`--expense`/`--transfer` each with a `-soft` background variant), then exposed as Tailwind utilities via a Tailwind v4 `@theme inline` block (`bg-canvas`, `bg-surface`, `bg-surface-header`, `bg-surface-alt`, `text-ink`, `text-ink-muted`, `border-edge`, `bg-accent`, `text-income`, `bg-income-soft`, etc.) — style components through these utility names, never hardcode a hex or a default Tailwind color (no more `text-red-600`/`bg-neutral-50`/etc. anywhere in the app). `income`/`expense`/`transfer` are semantic colors, deliberately a separate hue family from `accent` (the one used for primary actions/focus) — don't blur that distinction by reusing accent for a status meaning or vice versa.
  - **2026-07 contrast pass**: `--ground` was pulled noticeably further from `--surface` in both themes (tables were blending into the page background — a card sitting on a near-identical ground read as un-separated text, not a distinct surface). `--surface-header` (a table header row tint) and `--surface-alt` (zebra-row tint) are new tokens added for the same reason. `--border-strong` was darkened slightly alongside this pass since it's now also used for the header's bottom border.

- A thrown error inside a `startTransition(async () => { await action(...) })` callback does **not** reach an Error Boundary and is *not* automatically shown anywhere — it becomes a silent unhandled promise rejection unless caught explicitly. Every modal form dialog and delete button wraps its action call in try/catch (dialogs: `useState<string|null>` displayed near the buttons; delete buttons: `alert()`) — keep this pattern for any new Server-Action-calling client component, don't assume a thrown error will surface on its own.
- Money is stored as Postgres `numeric(12,2)`, which is exact fixed-point decimal — no floating-point rounding risk. Don't "fix" this into cents/bigint, it's already correct as-is.
- Derived/aggregate values (wallet balances, the VAT total on the Taxes page, `Total` on transactions) are **never stored** — always computed live from their inputs at read time. This was a deliberate choice over a maintained running counter: no increment/decrement logic to get right on every create/edit/delete/restore, and it can't ever drift. Keep following this for any future aggregate (don't add a stored counter without discussing it first).
- A `<dialog>` containing a `<form>` must never be nested *inside* another `<form>` — nested `<form>` elements are invalid HTML and get mangled by the browser's parser during SSR (the inner `<form>` tag is silently dropped since a form element is already open), causing a hydration mismatch. Where a modal-within-a-modal is needed (e.g. the "+ Add" entity button on the transaction form, which opens the entity-add dialog), render the inner `<dialog>` as a **sibling** of the outer `<form>` — a child of the outer `<dialog>`, not of the inner `<form>` — and trigger it via a ref from a plain button inside the outer form. See `transactions/transaction-form-dialog.tsx`.
- Money-affecting `numeric` columns (`net`, `vat_amount`, `rate`) come back from the Supabase JS client as **strings**, not numbers (same as the pre-existing `amount` column did) — always `Number(...)` them before doing arithmetic, same as `formatAmount`/`computeTotal` already do. The generated types in `database.types.ts` get this wrong (label them `number`) — see that Technical-section bullet, don't let the generated type's confidence override this.
- `net` and `vat_amount` are always non-negative magnitudes (DB-enforced via CHECK constraints) — direction/sign is never stored in the number itself, only implied by `type` (and, for transfers, by which wallet role a given wallet plays). Don't reintroduce signed `net` values; compute the signed effect at the point of use (wallet balance/ledger) instead, same as `wallets/queries.ts` already does.

- **Postgres RLS gotcha (learned the hard way, don't repeat it on future tables)**: an `UPDATE`'s resulting row must satisfy the table's `SELECT` policy too, not just the `UPDATE` policy's own `WITH CHECK` — this is enforced unconditionally, regardless of what the `UPDATE` policy's check says. So a SELECT policy that filters out `is_deleted = true` makes it *structurally impossible* for a user to ever set `is_deleted = true` on their own row (the resulting row would be invisible to their own write, which Postgres blocks outright — this produces the exact error `new row violates row-level security policy`). The fix: for any table with a soft-delete column, the SELECT RLS policy must stay unrestricted on that column (just `auth.uid() = user_id`, no `is_deleted` condition) — do the "hide deleted rows" filtering only in a shared app-level query helper. If accounts/categories/etc. get soft-delete later, follow the same pattern: don't add the deleted-flag condition to the table's SELECT policy.
