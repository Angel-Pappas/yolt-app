# Summary

This file exists so a fresh session can get up to speed on this app quickly. Keep it current: update the relevant section whenever a major feature is added, removed, or changed.

## General description

Yolt is a personal financial transactions tracker, built for a single user (the app owner only â€” no multi-tenant/sharing features are planned). It supports logging transactions (date, entity, wallet, description, net amount, VAT) behind email/password authentication, plus supporting lists of Entities, Wallets, and VAT rates, and a Taxes overview. Planned future features include financial projections, automatic generation of recurring transactions, and richer tax handling (multiple tax types, input/output VAT netting), plus whatever else comes up as the app grows.

## Main sections of the app

- **Authentication** â€” Signup, login, and logout. Email/password via Supabase Auth, including the email-confirmation flow. Pages: `/login`, `/signup`, `/auth/confirm` (confirmation link handler), `/auth/auth-code-error` (expired/invalid link fallback).
- **Home** (`/`) â€” Minimal landing page after login. Placeholder content for now.
- **Transactions** (`/transactions`) â€” Full CRUD on transactions (date, entity, wallet, description, net, VAT rate), all backed by RLS-scoped Server Actions (`actions.ts`: `addTransaction`, `updateTransaction`, `deleteTransaction`).
  - **Add**: a text "Add" button opens a modal (`transaction-modal.tsx`).
  - **Edit**: no button â€” clicking anywhere on a transaction's row (`transaction-row.tsx`, a client component) opens the same edit modal, pre-filled.
  - **Delete**: a trash-icon button per row (`delete-transaction-button.tsx`, `src/components/icons.tsx` for the icon â€” inline SVG, no icon library dependency); its click handler (and the dialog's) calls `stopPropagation()` so it doesn't also trigger the row's edit-open behavior. Delete is a **soft delete** â€” see the Database section below. Nothing is ever hard-deleted through the app.
  - Both modal contexts share one dialog implementation (`transaction-form-dialog.tsx`, native `<dialog>`, unique field ids via `useId`).
  - **Entity** field is a client-side-filtered search-and-select (`entity-combobox.tsx`) over the already-loaded entity list, with a hidden `entity_id` input and a "+ Add" button that opens the entity-add dialog (`entities/entity-form-dialog.tsx`) as a **sibling** of the transaction `<form>`, not nested inside it â€” nesting a `<dialog><form>` inside another `<form>` is invalid HTML and breaks under SSR. No inline creation from the combobox itself; the user must pick the newly-created entity from the search afterward.
  - **Wallet** and **VAT** are plain `<select>` dropdowns (populated from active wallets / active VAT rates).
  - **VAT Amount** and **Total** are calculated, read-only (live-updated client-side as Net/VAT change for display, but never trusted from the client): `vat_amount` is computed and stored server-side in the Server Action from `net Ă— the selected rate's current percentage` at save time (so editing a VAT rate's percentage later doesn't retroactively change past transactions' stored VAT amount). `Total` (`net + vat_amount`) is never stored at all â€” always computed on read via `computeTotal()` in `src/lib/format.ts`.
- **Entities** (`/entities`) â€” Full CRUD list of the counterparties transactions are made with (suppliers, customers, the state, etc.). Fields: name (required), VAT number (optional). Same soft-delete CRUD pattern as Transactions (row click opens edit modal, trash icon soft-deletes).
- **Wallets** (`/wallets`) â€” Full CRUD list of where money is held (bank accounts, cash, etc.). Fields: name only. No stored balance column â€” the list page shows each wallet's **current balance computed live** (`getWalletBalances()` in `wallets/queries.ts`, sums `net + vat_amount` across that wallet's active transactions), so it can never drift out of sync as transactions are added/edited/deleted/restored. Since the wallet name is a link (to the ledger view below) rather than the whole row, edit uses a pencil-icon button (`wallet-row.tsx`) instead of the row-click pattern used elsewhere.
  - **Ledger view** (`/wallets/[id]`) â€” a per-wallet transaction timeline with a running balance column, computed by walking that wallet's transactions in chronological order (`getWalletLedger()`) and then reversing for newest-first display, matching the rest of the app.
- **Taxes** (`/taxes`) â€” Read-only. Currently shows a single live-computed total (`SUM(vat_amount)` across active transactions, via `getTotalVat()`). Not a CRUD list â€” there's nothing to store, it's purely an aggregate over transactions. More tax types (and more intricacies â€” input/output VAT netting, filing periods, etc.) are expected here later.
- **Options** (`/options`) â€” Account settings (email/password, unchanged) plus a **VAT rates** section: full CRUD list (name, rate %) that populates the VAT dropdown on the transaction form. Same soft-delete pattern; files `vat-rate-*.ts(x)` alongside the existing `actions.ts`.
- All pages above share a common app shell (`src/app/(app)/layout.tsx`, a route group) with a nav bar (Home / Transactions / Entities / Wallets / Taxes / Options / Log out).

## Technical sections

- **Framework**: Next.js 16 (App Router, Turbopack), TypeScript, Tailwind CSS v4, ESLint. Scaffolded with `create-next-app`.
- **Data/validation libs installed for future use**: `@tanstack/react-query`, `zod` (not yet wired into any feature).
- **Auth/session plumbing**: `@supabase/ssr` with three client setups:
  - `src/lib/supabase/client.ts` â€” browser client
  - `src/lib/supabase/server.ts` â€” server client (Server Components/Actions)
  - `src/lib/supabase/proxy.ts` + `src/proxy.ts` â€” session-refresh and route protection. Next.js 16 renamed "middleware" to "proxy" (file `proxy.ts`, function `proxy`) â€” this app follows that convention. Auth check uses `supabase.auth.getClaims()` (not `getSession()`/`getUser()`), per current Supabase guidance.
  - Unauthenticated visitors are redirected to `/login` for any route except `/login`, `/signup`, and `/auth/*`.
- **Page structure**: `src/app/(app)/` is a route group holding the authenticated app shell â€” `layout.tsx` (nav bar), `page.tsx` (Home, `/`), `transactions/page.tsx` + `transactions/actions.ts` (`/transactions`), `options/page.tsx` (`/options`). Route groups (parens in the folder name) don't add a URL segment. `/login`, `/signup`, and `/auth/*` live outside this group (no nav bar).
- **Database**: Supabase Postgres. Project `yolt-app`, ref `mzfxfweljbfvyqlhvmzr`, region `eu-west-1`, free tier, org "Angel-Pappas's Org" (`yowsydnsxjwcujsjwtji` â€” same org as the Exalted-Character-App project).
  - Table `public.transactions`: `id` (uuid pk), `user_id` (uuid, defaults to `auth.uid()`), `date` (date), `description` (text), `net` (`numeric(12,2)`), `entity_id` (uuid, nullable FK â†’ `entities.id`, `on delete set null`), `wallet_id` (uuid, nullable FK â†’ `wallets.id`, `on delete set null`), `vat_rate_id` (uuid, nullable FK â†’ `vat_rates.id`, `on delete set null`), `vat_amount` (`numeric(12,2)`, not null default 0 â€” computed and stored server-side at save time, see Transactions section above for why), `created_at`, `is_deleted` (default false), `deleted_at` (nullable). There is **no `amount` column anymore** â€” it was replaced by `net` + `vat_amount` (with `Total` derived, never stored). RLS enabled, all policies scoped to `auth.uid() = user_id`.
  - Tables `public.entities`, `public.wallets`, `public.vat_rates`: same shape/conventions â€” `id`, `user_id` (default `auth.uid()`), `name`, plus one extra field each (`vat_number` for entities, `rate numeric(5,2)` for vat_rates; wallets has no extra field), `created_at`, `is_deleted`, `deleted_at`. Same RLS pattern as transactions.
  - **Soft delete**: the app's "Delete" action does `update is_deleted = true, deleted_at = now()`, never a real `DELETE` â€” on **every** table (transactions, entities, wallets, vat_rates). Filtering out `is_deleted = true` rows is done **only at the app-query level**, via a single shared helper per table (`getActiveTransactions()`, `getActiveEntities()`, `getActiveWallets()`, `getActiveVatRates()`) â€” see the important RLS gotcha in Notes to myself: each table's SELECT RLS policy deliberately does *not* filter on `is_deleted`, because it can't without breaking the soft-delete UPDATE itself. There is **no DELETE policy** on any of these tables (dropped/never added deliberately) â€” the app has no way to hard-delete a row, by design. Only the user (owner) can ask for a permanent delete or a restore, done by directly running SQL via the Supabase MCP (`execute_sql`/`apply_migration`), not through any in-app feature.
  - Editing a `vat_rates` row's `name`/`rate` only affects the dropdown label and future transactions â€” it never retroactively changes `vat_amount` on transactions that already used that rate, since that value was snapshotted at save time.
- **Hosting/deploy**: Vercel, connected to GitHub repo [Angel-Pappas/yolt-app](https://github.com/Angel-Pappas/yolt-app), auto-deploys on push to `main`. Env vars (`NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY`) are set directly in the Vercel dashboard (mirrors `.env.local`, which is gitignored; `.env.example` is committed as a template).

## Formatting conventions

Greek-style formatting is used everywhere the app displays dates/numbers: dates as `dd/mm/yyyy`, numbers with `.` as the thousands separator and `,` as the decimal separator, always rounded to exactly 2 decimals. Shared helpers live in `src/lib/format.ts` (`formatDate`, `formatAmount`, `computeTotal`) â€” use these instead of ad-hoc formatting/arithmetic whenever displaying a date or amount anywhere in the app. Native HTML form inputs (`<input type="date">`, `<input type="number">` in the add-transaction modal) are exempt â€” their `value` must stay ISO/period-decimal per the HTML spec, only their on-screen picker rendering is browser/OS-locale-dependent, and that isn't something the app controls.

## Notes to myself

- This is a single-user app. Don't add roles/teams/sharing/multi-tenant complexity unless explicitly asked â€” but keep RLS correct regardless, it's good practice regardless of user count.
- Money is stored as Postgres `numeric(12,2)`, which is exact fixed-point decimal â€” no floating-point rounding risk. Don't "fix" this into cents/bigint, it's already correct as-is.
- Derived/aggregate values (wallet balances, the VAT total on the Taxes page, `Total` on transactions) are **never stored** â€” always computed live from their inputs at read time. This was a deliberate choice over a maintained running counter: no increment/decrement logic to get right on every create/edit/delete/restore, and it can't ever drift. Keep following this for any future aggregate (don't add a stored counter without discussing it first).
- A `<dialog>` containing a `<form>` must never be nested *inside* another `<form>` â€” nested `<form>` elements are invalid HTML and get mangled by the browser's parser during SSR (the inner `<form>` tag is silently dropped since a form element is already open), causing a hydration mismatch. Where a modal-within-a-modal is needed (e.g. the "+ Add" entity button on the transaction form, which opens the entity-add dialog), render the inner `<dialog>` as a **sibling** of the outer `<form>` â€” a child of the outer `<dialog>`, not of the inner `<form>` â€” and trigger it via a ref from a plain button inside the outer form. See `transactions/transaction-form-dialog.tsx`.
- Money-affecting `numeric` columns (`net`, `vat_amount`, `rate`) come back from the Supabase JS client as **strings**, not numbers (same as the pre-existing `amount` column did) â€” always `Number(...)` them before doing arithmetic, same as `formatAmount`/`computeTotal` already do.
- The 2 pre-existing test transactions (before entities/wallets/VAT existed) were backfilled with `net = amount`, `vat_amount = 0`, and `entity_id`/`wallet_id`/`vat_rate_id` left `null` â€” they'll show "â€”" for those columns. That's expected, not a bug.
- Next.js 16 has real breaking changes vs. training-data knowledge (biggest one: `middleware.ts` â†’ `proxy.ts`). The bundled docs at `node_modules/next/dist/docs/` and Supabase's own docs (via the Supabase MCP `search_docs` tool) are the source of truth â€” check them before assuming an API when working with newer package versions.
- This dev machine (Sofoklis's Windows box) didn't have Node.js or the GitHub CLI installed originally; both were installed via `winget`. `gh` is installed but **not authenticated** â€” git push/pull works through Git Credential Manager (`credential.helper=manager`), which is already authenticated, not through `gh`.
- No Vercel CLI/token is set up locally. Vercel project settings (env vars, redeploys) go through the user in the dashboard â€” not automatable from a coding session right now.
- PowerShell tool calls in this environment don't inherit PATH updates made mid-session (e.g. after a `winget install`). If a newly installed CLI isn't found, refresh with:
  `$env:Path = [System.Environment]::GetEnvironmentVariable("Path","Machine") + ";" + [System.Environment]::GetEnvironmentVariable("Path","User")`
  at the start of the same PowerShell command block.
- User's standing rules (see also memory): never use any email address (including the user's own) without asking first â€” use `example@example.com` for placeholders/tests. Don't drive a browser to visually test changes â€” verify via build/lint/typecheck only and let the user do functional/visual testing.
- Sibling project: `Exalted-Character-App` (Vite + React SPA, different stack) â€” not related to this app beyond sharing the same Supabase org and GitHub account.

- Soft-deleted transactions are invisible everywhere in the app by design â€” there's no "trash/recover" UI. To restore or permanently purge one, the user asks directly (in chat) and it gets done via SQL against the `transactions` table, not through a feature.
- **Postgres RLS gotcha (learned the hard way, don't repeat it on future tables)**: an `UPDATE`'s resulting row must satisfy the table's `SELECT` policy too, not just the `UPDATE` policy's own `WITH CHECK` â€” this is enforced unconditionally, regardless of what the `UPDATE` policy's check says. So a SELECT policy that filters out `is_deleted = true` makes it *structurally impossible* for a user to ever set `is_deleted = true` on their own row (the resulting row would be invisible to their own write, which Postgres blocks outright â€” this produces the exact error `new row violates row-level security policy`). The fix: for any table with a soft-delete column, the SELECT RLS policy must stay unrestricted on that column (just `auth.uid() = user_id`, no `is_deleted` condition) â€” do the "hide deleted rows" filtering only in a shared app-level query helper. If accounts/categories/etc. get soft-delete later, follow the same pattern: don't add the deleted-flag condition to the table's SELECT policy.
